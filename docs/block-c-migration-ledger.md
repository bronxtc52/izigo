# Block C — Migration Ledger (резерв таймстампов миграций)

Цель: развести таймстампы миграций 7 фич Блока C, чтобы параллельные ветки
**не коллизились** и порядок применения был детерминирован.

Все миграции Блока C живут в `mh-calc-backend-main/Modules/Calculator/Database/Migrations`
(там же, где остальные миграции модуля Calculator).

## Последняя существующая миграция (на момент scaffold, 2026-06-22)

`2026_06_22_040000_create_member_agreement_acceptances_table.php`

Все диапазоны ниже выбраны **строго позже** неё (042xxx → 055xxx).

## Правило (для всех веток Блока C)

1. **Каждая фича создаёт миграции ТОЛЬКО в своём диапазоне** из таблицы выше.
   Внутри диапазона есть запас (xx = 00..99) под дополнительные миграции фичи.
2. Не «перепрыгивай» в чужой диапазон, даже если он свободнее — иначе merge-train
   потеряет детерминизм порядка.
3. Имена файлов держим как в таблице (можно добавлять новые в своём диапазоне).
4. CI миграции не гоняет — `migrate` идёт в `docker/start.sh` (forward-only,
   см. память `izigo-ci-migrations-startsh`). Любое расширение (`ltree` и т.п.)
   разрешать до push.
5. Колонки PII (`telegram_username`, `payout_details`, KYC) новые миграции **не
   дублируют** — C5 читает существующие.

## Фронт-API: правило непересечения (webApi.js / miniapp api.js)

`webApi.js` и `miniapp/api.js` **не дробим** (рискованно). Каждая фронт-фича
добавляет свои вызовы **в КОНЦЕ** соответствующего файла, в секции, обрамлённой
маркерами:

```js
// >>> Block C <feature>
export const ... = ...;
// <<< Block C <feature>
```

Так merge-train разрешает конфликты тривиально (каждая фича = свой блок в хвосте).
