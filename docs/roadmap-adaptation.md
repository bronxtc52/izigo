# Роадмап адаптации: «Лидер» → binar-mlm (IziGo)

> Источник идей — [`handoff-leader-new.md`](./handoff-leader-new.md). Здесь — **что и в каком
> порядке строим у себя**, с привязкой к существующему коду. Scope согласован: два блока —
> **A. Отчёты/аналитика** и **B. Админ-дерево/плейсмент + онбординг**. Остальные блоки «Лидера»
> (news, kb, chat, voip, subdivisions) — отложены; мультивалюта, доп.шлюзы, password/2FA/email,
> «mining»-механизм — исключены навсегда.

## Незыблемые рамки (см. memory `izigo-core-constraints`)
1. **Движок бонусов не трогаем.** Всё ниже — `read-over-engine`: только читаем выходы
   (`MemberEarning`, `MemberBonusLine`, `LedgerEntry`), математику `Modules/Calculator/Domain/*`,
   `Services/Bonus/*`, `Services/Rank/*`, `CalculatorService.php` НЕ меняем.
2. **USD-only.** Без валютных полей/курсов.
3. **Telegram-only** для платежа (TON Pay) и авторизации (initData / Login Widget).

Каждый PR в рамках роадмапа должен проходить чек: `git diff` не задевает `Modules/Calculator`,
не вводит валюту/шлюзы/пароли.

---

## Блок A — Отчёты / аналитика

**Что переиспользуем (бэкенд):** `Modules/Calculator/Services/AdminReportService` (расширить
read-методами), `LedgerService`, `WalletService`; модели `MemberEarning`, `MemberBonusLine`,
`LedgerEntry`, `Order`. **Фронт:** веб-админка `src/views/admin/web/*` (паттерн Antd + `webApi.js`,
Bearer Sanctum — см. memory `izigo-admin-in-web-not-miniapp`, `izigo-web-admin-api-positional-args`).

### A1. Админ-отчёты (новый экран `Reports`)
- Под-отчёты: **Балансы** (по участникам: available / held / clawback-debt), **Пользователи**
  (ранг/статус/PV/объём/актив), **Продажи** (по периоду), **Расход на бонусы** (по типам).
- Фильтр по периоду (date-range) + **экспорт CSV**.
- Бэкенд: новые read-эндпоинты в `AdminReportService` поверх `LedgerService`/`WalletService`;
  агрегаты считаем по уже сохранённым выходам движка, **не пересчитывая бонусы**.
- Фронт: `src/views/admin/web/Reports.js` + пункт меню в `WebAdminShell.js`, gate по роли
  (owner/finance), функции в `webApi.js`.

### A2. Выписки участнику (Mini App)
- В Income-вкладке (`src/views/miniapp/MiniAppShell.js`) под-экран «Выписка»: начисления по
  периодам, фильтр по типу бонуса (binary/referral/leader/rank), история «прогонов», экспорт.
- Поверх `MemberEarning` / `MemberBonusLine` (read-only). Суммы в USD.

**Тесты (риск средний — деньги/контракты чтения):** API-контракты отчётов (агрегаты совпадают с
суммой строк леджера), negative-cases доступа (чужая роль / не-owner → 403). Golden-сверка: сумма
выписки участника = сумма его `MemberEarning` за период.

## Блок B — Админ-дерево / плейсмент + онбординг

**Что переиспользуем:** `StructureService` (чтение дерева), `Placement/*` (auto-spillover +
manual стратегии — уже есть), `AdminAuditLog`, `src/views/admin/web/Users.js` + `MemberCard.js`.

### B1. Админ-генеалогия (view)
- Визуальное бинарное дерево: левая/правая нога, drill-down в карточку участника
  (`MemberCard.js`). Поверх `StructureService` (только чтение). Без изменения структуры.
- Фронт: расширить `Users.js` или новый `Genealogy.js` в `admin/web/`.

### B2. ⚠️ Ручной плейсмент / перенос участника (HIGH-RISK)
- UI-действие «поставить/перенести участника под спонсора/ногу», опираясь на существующие
  `Placement/*`.
- **Радиус поражения большой:** перемещение в дереве меняет ВХОДНЫЕ данные движка → влияет на
  будущие расчёты (сам код движка не трогаем, но его вход — да). Поэтому **обязательно**:
  - admin-only (owner), запрет для прочих ролей;
  - запись в `AdminAuditLog` (кто/кого/откуда-куда/когда);
  - **подтверждение + preview эффекта (dry-run)** до применения;
  - идемпотентность и возможность отката (логировать прежнюю позицию).
- Это единственный пункт роадмапа, меняющий данные, а не только читающий. Делать через рабочую
  ветку + PR/self-review (см. git-дисциплину), не прямым push в main.

### B3. Онбординг / юр.согласия
- Шаг акцепта условий при первом входе (Telegram-онбординг, без email/паролей): показать текст
  соглашения → зафиксировать факт акцепта (версия + дата).
- Админ управляет текстом соглашения (хранение версии). Аналог «Лидер»-Agreement, но без
  email-подтверждений.

### B4. Насыщенный профиль / дашборд участника
- Выделенный **Dashboard**-таб участника: личные KPI (доход по типам, статус, прогресс к рангу).
- Полный **профиль**: реквизиты вывода (TON-адрес — см. memory
  `izigo-withdrawal-payout-details-is-ton-address`), KYC-статус, реферальная, статус соглашений.

**Тесты:** для B2 — обязательны negative-cases (не-owner не может двигать; нельзя поставить узел
сам под себя/в свой downline — цикл; аудит пишется). Для B3 — факт акцепта сохраняется и блокирует
повторный показ.

## Порядок выполнения (предложение)
1. **A1 Админ-отчёты** — чистый read, ценность сразу, низкий риск. Старт.
2. **A2 Выписки участнику** — read, дополняет тонкую Income-вкладку.
3. **B1 Админ-генеалогия (view)** — read, визуальная база для B2.
4. **B4 Профиль/дашборд** — read, наполняет UI.
5. **B3 Онбординг/согласия** — небольшая запись (факт акцепта), низкий риск.
6. **B2 Ручной плейсмент** — ПОСЛЕДНИМ: самый рискованный, нужен dry-run/preview + аудит + PR.

- **Исключено навсегда (🔴, ограничения):** мультивалюта/курсы, доп. платёжные шлюзы,
  password/2FA/email-auth, voip, «mining» как отдельный механизм начислений, reward_points-шоп.
