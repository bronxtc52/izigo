# Продакшн-ревью платформы IziGo — 2026-07-02

**Метод:** 6 параллельных ревью-агентов (security, денежный контур, AI-ассистент + Блок C,
фронтенд, ops/инфра, фактический прогон тестов). Все 4 денежных P1 дополнительно подтверждены
построчным чтением кода. Ревью статическое, без динамической эксплуатации находок.

**Вердикт:** платформа готова к **бете под присмотром**, но НЕ готова к безнадзорному
продакшену. P0-блокеров нет ни в одном контуре. Тесты зелёные (backend 350/350, 1239 assertions;
фронт lint 0 ошибок + `next build` успех, 26 страниц; бот 6/6). Security-ядро реализовано
корректно. Найдено **11 P1**, из них четыре — в денежном контуре с прямыми сценариями
потери/задвоения денег. Все фиксы локальные и дешёвые.

**Дальнейшая судьба находок:** ТЗ на закрытие — `docs/specs/2026-07-02-p1-production-hardening.md`
(скоуп: 11 P1 + 3 дешёвых P2; решение Гейта 0).

## P1 — чинить до широкой раскатки

### Деньги (все 4 подтверждены построчно)

| # | Находка | Сценарий | Где |
|---|---|---|---|
| 1 | Нет try/catch в цикле поллинга | Один «отравленный» платёж → исключение валит весь прогон: **все подтверждения и TTL встают навсегда**, деньги висят на merchant-кошельке | `PaymentService.php:202-212` |
| 2 | Гонка параллельных recompute | Два платежа подтверждаются одновременно (поллер + ручной `payments/{id}/check`) → дельта от неполного снимка → **повторное начисление в кошельки** | `ActivationService.php:128-194` |
| 3 | Autoship: списание закоммичено, активация вне транзакции | Сбой между charge и activate → **деньги списаны, активация потеряна навсегда** (ретрай — no-op по идемпотентным ключам) | `AutoshipService.php:139-153` |
| 4 | TTL-экспирация слепа к ошибкам опроса | Аутедж toncenter или оплата позже 24ч → реально оплаченный платёж помечен `expired`, никто его больше не опрашивает — **тихая потеря денег юзера** | `PaymentService.php:219-225` |

## Самые важные P2

- **Нет liveness-probe** на ACA: смерть `schedule:work` = TON-поллинг встал, платежи не
  подтверждаются, контейнер «здоров» — самый тихий сценарий отказа.
- **Prototype pollution** через dot-ключи i18n-оверрайдов на фронте (`__proto__.x` пишет в
  глобальный прототип, эндпоинт публичный).
- После таймаута поллинга TonPay кнопка «Оплатить» возвращается → возможен второй on-chain
  перевод по тому же memo.
- `GET /admin/members/{id}` отдаёт `telegram_username` ролям finance/support/leader — обход
  C5-маскирования (единственная фактическая PII-утечка, внутренним ролям).
- Бот: нет `bot.catch()` (crash-loop от бытовой ошибки) + деплой вручную по `:latest` мимо CI.
- Окно матчинга 100 переводов без пагинации — при росте оборота платежи начнут выпадать.
- Гонка «лид платит в момент экспирации»: фильтр не учитывает платежи в статусе `created` →
  осиротевший платёж становится «ядом» для находки №1.
- Sanctum-токен админки: нет logout/отзыва (утёкший bearer живёт 12ч); окно replay initData 24ч;
  заход лида на member-only эндпоинты Блока C даёт грязный 500 вместо 403.
- i18n-оверрайды не знают локали `en` (Mini App двуязычный с PR#17); unmasked PII-экспорт
  аудируется best-effort (в отличие от fail-closed reveal); нет троттла 30 msg/sec к Telegram;
  outbox — at-least-once (дубль при падении процесса — trade-off, не задокументирован).
- Next.js 14.2.15 с известными CVE (обновление дешёвое); `NEXT_PUBLIC_SERVER_PROD='false'` —
  truthy-строка, GA/Метрика грузятся и в dev; webvisor Метрики пишет сессии финансового
  приложения третьей стороне.
- Бэкап Postgres: retention 7д, geo-redundancy off, restore-runbook нет.

## Что подтверждено как «чисто»

- **Деньги-математика:** везде int-центы, float-арифметики нет, memo-матчинг точный
  (pay:5 ⊄ pay:55), переплата принимается, ledger с двойной записью и `assertBalanced`,
  баланс в минус не уходит, лид-грабля (backfill до delete) закрыта.
- **Security:** initData/Login Widget HMAC по канону, IDOR-скоуп консистентный по всем
  cabinet-эндпоинтам, RBAC deny-by-default, PII-reveal fail-closed, SQL параметризован,
  mass-assignment whitelist, секретов в коде и бандле нет.
- **AI-ассистент:** образцовый — флаг на бэке, rate limit по member_id, вопрос отдельным
  user-message (инъекция в system невозможна), PII наружу не течёт; готов к включению флага
  как есть (ANTHROPIC_API_KEY на ACA прокинут, проверено).
- **Блок C:** outbox с lock-паттерном, helpdesk скоупит «только свои», PII-reveal fail-closed,
  санитайзер Telegram-HTML по глобальному правилу.
- **Инфра:** OIDC, все секреты keyvaultref, APP_DEBUG=false, логи в Log Analytics, все ревизии
  Healthy, maxReplicas=1 фиксирует single-replica допущения, Postgres PITR 7 дней.

## Рекомендуемый порядок фиксов

1. **Пакет «деньги»** (№1–4): try/catch per-payment, advisory-lock в activate, транзакция вокруг
   autoship, «не экспирировать при ошибке опроса». Ветка + PR (auth/деньги — рисковая зона).
2. **Быстрые победы ops** (№8, 9, bot.catch): убрать `|| true`, test-job перед деплоем.
3. **Флаги Блока C на бэк** (№5) + TON-чексумма на фронт и бэк (№7).
4. **Sentry на фронт + error boundaries** (№10, 11) — до включения флагов.
5. Остальные P2 — отдельной уборкой.
