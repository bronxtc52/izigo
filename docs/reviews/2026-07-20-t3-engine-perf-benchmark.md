# t3 — Перф-бенчмарк движка пересчёта: результаты и рекомендации (2026-07-20)

**Задача:** measurement-first замер производительности движка пересчёта бонусов на
воспроизводимом синтетическом дереве, БЕЗ внесения оптимизаций (ядро `Domain/**`, `V2/**`
не тронуто ни строкой — diff по запретной зоне пуст). Инструмент — новая artisan-команда
`calculator:bench-engine` (обвязка: генератор дерева + DB::listen-агрегатор + отчёт).

## Методика

- **Дерево:** детерминированный генератор (`Support/Bench/SyntheticTreeGenerator`) —
  почти полное бинарное дерево по heap-нумерации, fixed seed 20260720, пакеты ~50/30/20,
  спонсор — случайный предок; инварианты members соблюдены (единственный корень,
  unique(parent_id,position), ltree path, id родителя/спонсора < id ребёнка). Вставка —
  чанковые bulk-insert по 1000. Реальная форма прод-сети неизвестна — допущение зафиксировано.
- **Фазы V1:** `v1_load` — `EloquentNetworkRepository::load()`; `v1_engine` — чистый
  `CompensationEngine::calculate()` (read-only, 0 SQL); `v1_activate` — полный
  `ActivationService::activate()` (новый idempotency_key + смена package_id листа →
  полный recompute + перезапись снапшота + ledger-дельты).
- **Фазы V2:** `v2_mark_paid` — боевой путь оплаты `OrderService::markPaid()` в транзакции
  (V1 activate + `PaidOrderV2Pipeline` под флагами volumes/statuses/referral/awards);
  `v2_structure_bonus` — разовый `v2:structure-bonus:run` текущего half-month окна.
- **Метрики:** wall — `hrtime()`; SQL count/время — `DB::listen`-агрегатор (НЕ
  enableQueryLog); память — `memory_reset_peak_usage()` + `memory_get_peak_usage(true)`
  на фазу; топ-10 нормализованных SQL на размер.
- **Прогон:** `--fresh --sizes=1000,5000,20000 --iterations=3 --engine=v1,v2`, warmup
  перед замерами; БД `izigo_bench` (migrate:fresh на каждый размер). Числа в таблицах —
  среднее из 3 итераций (min…max в скобках у wall).
- **Контроль методики:** после каждого размера чистое ядро на финальной сети сверено с
  персистом: `engine_total == sum(member_earnings) == нетто ledger-проводок расхода` —
  **сошлось на всех трёх размерах** (5 900 527 ¢ / 34 960 637 ¢ / 161 078 818 ¢).

## Окружение (hardware зафиксирован)

- VM mh-central (Azure): **AMD EPYC 9V45**, 8 vCPU (4 cores × 2 SMT), **62 GiB RAM**
  (`lscpu`/`free -h`); Linux 6.17.0-1020-azure.
- PHP **8.3.31** (образ `izigo-php-dev`, докер, `--network host`), Laravel **12.63.0**.
- PostgreSQL **16.14** (`izigo-test-pg`, postgres:16-alpine, 127.0.0.1:5544), БД `izigo_bench`.
- Замер под `-d memory_limit=2048M` (см. находку §OOM ниже). Это dev-VM, не ACA — абсолютные
  миллисекунды на прод-контейнере будут отличаться, тренды и профиль SQL — нет.

## Анализ

1. **Ядро движка — квадратичное по N.** Чистый `CompensationEngine::calculate` (0 SQL,
   чистый CPU): 0.245 с @1k → 6.4 с @5k → 108 с @20k. Фит `t ≈ k·N²` (k≈0.25 мкс) даёт
   6.1 с и 98 с — совпадение в пределах шума. Это in-memory сложность самого расчёта
   (проходы вверх по цепочкам предков на каждый узел), НЕ база.
2. **Персист снапшота — линейный, но построчный.** SQL на полный activate ≈ **12.4·N**
   запросов (8.7k @1k → 247.5k @20k). Лидер с огромным отрывом — построчный
   `MemberBonusLine::create()` (~10.8 строк на участника на каждый recompute, 393.9 с
   суммарного SQL-времени на размере 20k), затем безусловный `update members.rank_id`
   по КАЖДОМУ узлу (даже без изменения ранга) и построчный `MemberEarning::create()`.
3. **Полный activate = ядро + персист.** @20k: 205 с wall ≈ 108 с CPU ядра + 74 с SQL +
   накладные. Обе составляющие растут быстрее, чем хотелось бы: CPU — O(N²), SQL — O(N)
   с большой константой.
4. **V2-надстройка пост-оплаты — дёшево.** `v2_mark_paid` ≈ `v1_activate` (разница в
   пределах шума): стоимость пути оплаты почти целиком — V1-recompute, шаги
   volumes/statuses/referral/awards добавляют десятки запросов, не тысячи.
   ⚠️ Ограничение методики: `v2_structure_bonus` (7 мс) нерепрезентативен — до cutover
   PV-лоты копятся только через V2-пайплайн, у bench-дерева лоты есть лишь от заказов
   самого бенча; замер полного матчинга большой сети — после наполнения v2_pv_lots.
5. **🔴 OOM-находка (операционный риск уже сейчас):** при дефолтном `memory_limit=128M`
   (php:8.3-cli — ровно то, что в прод-Dockerfile бэка, override отсутствует) первый
   полный прогон **упал fatal'ом `Allowed memory size … exhausted` на size=5000** в
   `CalculationResult::addBonus`. Под 2048M пиковые значения фаз: 46 / 70 / 164 MB
   (1k/5k/20k), т.е. сам расчёт живёт скромно, но совокупная аллокация процесса при
   5k+ участников упирается в 128M. Порог фатала — между 1k и 5k участников.

## Вердикт и рекомендации

- **Сейчас оптимизация НЕ нужна.** На текущем масштабе сети (сотни участников) полный
  recompute — доли секунды/единицы секунд, поллер тикает раз в минуту, глобальный
  advisory-lock не является узким местом.
- **Порог деградации — ~3–5k участников.** Каждый оплаченный заказ = полный recompute
  под глобальным локом: @5k это ~28 с на покупку (одна покупка в полминуты насыщает
  систему и подвешивает конкурентные активации на локе), @20k — ~3.4 мин (неработоспособно).
  Мониторить рост members; при приближении к ~2–3k — планировать волну оптимизации.
- **Что чинить в первую очередь (по данным, в порядке отдачи):**
  1. `memory_limit` прод-контейнера поднять (512M — дёшево и снимает fatal-порог ~5k)
     — единственный пункт, который стоит сделать не дожидаясь роста;
  2. батч-вставка `member_bonus_lines`/`member_earnings` чанками вместо построчного
     `create()` — крупнейшая SQL-статья (десятки × меньше запросов), чистая обвязка
     `ActivationService::recompute`, ядро не трогается;
  3. `members.rank_id` обновлять только при фактическом изменении ранга (сейчас N
     no-op UPDATE на каждый recompute);
  4. дальняя перспектива — инкрементальный пересчёт вместо полного (уже помечен в
     докблоке `ActivationService` как «позже»): единственное лекарство от O(N²) ядра,
     но это изменение ядра — отдельное решение владельца.
- **Octane НЕ поможет.** Профиль стоимости — CPU квадратичного ядра (in-process, 0 SQL)
  + линейный построчный SQL-персист. Octane убирает bootstrap фреймворка на запрос
  (~десятки-сотни мс) — против 27–205-секундного recompute это <1%. Числа бенча это
  фиксируют: узкое место не в подъёме приложения.

## Воспроизведение

```bash
# из корня репо; БД izigo_bench на izigo-test-pg (5544) должна существовать
docker run --rm --network host -v $PWD/mh-calc-backend-main:/app \
  -e DB_CONNECTION=pgsql -e DB_HOST=127.0.0.1 -e DB_PORT=5544 \
  -e DB_USERNAME=postgres -e DB_PASSWORD=postgres -e DB_DATABASE=izigo_bench \
  izigo-php-dev php -d memory_limit=2048M artisan calculator:bench-engine \
  --fresh --sizes=1000,5000,20000 --iterations=3 --engine=v1,v2 \
  --json=/app/storage/bench-results.json --md=/app/storage/bench-results.md
```

Гарды A-t3: `migrate:fresh` только при явном `--fresh`; production → отказ; allowlist БД
`izigo_bench | izigo_test*`; непустая `members` без `--fresh` → отказ. Команда не
регистрируется в prod-контейнере и не стоит в расписании; в CI гоняется только smoke
(~50 узлов) внутри `php artisan test`.
