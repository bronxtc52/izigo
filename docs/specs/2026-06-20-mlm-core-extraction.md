# ТЗ: Фаза 1 / цикл 1 — извлечение чистого доменного ядра (PV)

**Дата:** 2026-06-20
**Основание:** `docs/specs/2026-06-20-izigo-platform-v1.md`, `roadmap.md`.
**Статус:** на согласовании (Гейт 1).

## Цель

Вынести компенсационную логику калькулятора в **чистое тестируемое доменное ядро без
Laravel-зависимостей**, перевести базу бонусов **BV→PV** (модель ТЗ), покрыть **golden
unit-тестами**. Это фундамент биллинга — приоритет на корректность.

## Сценарии (для кого)
- Будущая платформа вызывает ядро для реал-тайм начисления бонусов (детерминированно, без БД).
- Разработчик меняет проценты/пороги через конфиг, не трогая код формул.
- Тесты служат исполняемой спецификацией маркетинг-плана.

## Решения
- **Калькулятор-витрина НЕ трогаем** (работает на BV, :8000/:3001). Ядро — новое, параллельно.
  Перевод витрины на ядро — отдельный цикл.
- **Деньги и PV — целочисленные minor units** (USD-центы; PV в сотых), без float. Value Objects.
- Расположение: `Modules/Calculator/Domain/**` (чистый PHP), тесты `Tests/Unit/Domain/**` (без БД).
- Справочники (Rank/Package/проценты) — через **интерфейсы-репозитории** + конфиг, инъекцией.
- i18n/форматтеры — вне ядра (ядро возвращает данные, не локализованные строки).

## Содержание ядра (перенос логики 1:1, база BV→PV)
- **Бинарный бонус:** пайринг по малой ветке (min двух ног) в PV, carryover/flush спаренного
  объёма, % вверх по бинар-родителям. Настраиваемый процент (дефолт 5%) + задел под cap.
- **Лидерский:** bonus-on-bonus (% от бинарного) по линии спонсоров, глубина = maxLevel,
  матрица `[level][package][rank]`, rank-compression `MAX_RANK_DIFF=2`.
  **Фикс бага** `BonusLeaderService.php:99` (null-deref на цепочке спонсоров до корня).
- **Реферальный:** % от PV пакета по линии спонсоров, глубина 2, проценты по пакету.
- **Ранговый:** разовая выплата при достижении ранга (сумма из справочника).
- **Квалификация рангов:** конъюнкция (малая ветка PV ≥ порог + кол-во ЛП + ЛП в ранге),
  темпоральная отсечка по `maxNodeId` (ранг «на момент» постановки узла).
  **Фикс корректности (как в бинаре):** малая ветка для квалификации = min ПО ДВУМ ногам
  (отсутствующая = 0) → нужны обе ноги. Исходник (`NodeForCheckRanks`) брал min по
  присутствующим детям и квалифицировал по одной ноге. Покрыто `RankSnapshotTest`.
- **Модель дерева:** placement (бинар) + sponsorship (ЛП), накопление PV вверх.

## PV-модель (из ТЗ)
- BV убран. 1$ = 0.9 PV. Пакеты: Bronze 90 PV / Silver 180 PV / Gold 540 PV (цены $100/$200/$600).
- Бонусы = % от PV. Деньги — USD, 1 PV = $1 (для выплаты).

## Тесты (golden)
- Чистые unit-тесты (PHPUnit, без БД): пайринг с carryover (3 шага), реферальный по уровням,
  лидерский с compression, квалификация рангов до верхнего (структуры 7/14/36 узлов).
- Портировать ручные пересчёты из `Tests/Unit/StructureTest::testMarketing2/testRank*`,
  пересчитав ожидаемые значения под PV (90/180/540).
- Тест на баг-кандидат: цепочка спонсоров до корня не роняет лидерский расчёт.

## Границы (НЕ в этом цикле)
- Глобальный бонус (пул 3%, периодический) — отдельно (не реал-тайм).
- Нормализованная генеалогия (ltree/closure), реальные члены сети, API/кабинет — следующий цикл.
- Перевод калькулятора-витрины на ядро.
- Cap бинара — заложить параметр, реализацию/значение — позже.

## Приёмка
- Пакет `Modules/Calculator/Domain` без `use Illuminate\*`/`__()`/Eloquent в формульных классах.
- `php artisan test --testsuite=Unit` (или прямой phpunit) — golden-тесты ядра зелёные, без БД.
- Калькулятор-витрина по-прежнему работает (:8000/:3001), тесты LocalAuthTest зелёные.
