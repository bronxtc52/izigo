# ТЗ: Фаза 1 цикл 2 + Фаза 2 — реальная сеть + кабинет партнёра + админка

**Дата:** 2026-06-20
**Основано на:** `docs/specs/2026-06-20-izigo-platform-v1.md`, `roadmap.md`, `plan.md`,
доменное ядро `Modules/Calculator/Domain` (Фаза 1 цикл 1 — готово).
**Статус:** на согласовании (Гейт 1).

## 1. Цель

Перевести IziGo из режима «симулятор» в платформу с **реальной сетью**: реальные
партнёры (members), нормализованная генеалогия (Postgres `ltree`), расчёт через
извлечённое доменное ядро на **живых данных**. Поверх этого — **кабинет партнёра**
(доход/дерево/прогресс рангов) и **админка MVP с полным RBAC**. Каналы: web +
Telegram Mini App + бот.

Деньги (ledger/кошелёк/выплаты) и реальная оплата — **вне этого цикла** (Фазы 3–4).
Активация пакета — мок (самообслуживание), чтобы кормить движок событиями.

## 2. Зафиксированные решения (Гейт 1, опрос)

| Параметр | Решение |
|---|---|
| Объём цикла | **Фундамент данных + кабинет + админка** (вариант A: сначала реальная сеть, затем UI поверх живых данных) |
| Активация пакета | **Самообслуживание, мок-оплата** — кнопка «оплачено» → идемпотентное событие → `CompensationEngine`. Деньги не двигаются |
| Размещение в дереве | **Оба режима** (авто-спилловер в слабую ногу BFS + ручной выбор слота) + переключатель компании, через `PlacementStrategy` |
| Каналы | **web + Telegram Mini App + Telegram-бот** |
| RBAC | **Полный каркас 4 ролей** (владелец/финансы/лидер/саппорт) + permission-гейты |
| БД | **PostgreSQL** (в т.ч. локально — `ltree` только Postgres) |
| Стартовые данные | С нуля + сид для демонстрации |

> ⚠️ **Размер.** Объём = Ф1ц2 (данные) + Ф2 (кабинет+админка) + кусок Telegram-фазы.
> Это не «тонкий срез». На Гейте 2 (план) разбиваем на упорядоченные под-этапы
> (вертикальный срез сначала, каналы/полнота — следом) и режем по факту.

## 3. Акторы и сценарии (user stories)

- **Партнёр** регистрируется по реф-ссылке (web или Telegram deep-link), привязывается
  к аккаунту (auth есть), **ставится в бинар-дерево** (по режиму компании), **активирует
  пакет** (мок-оплата) → видит в кабинете **реальный доход** (разбивка по бонусам + логика
  расчёта), **дерево команды**, **прогресс рангов**, реф-ссылку.
- **Спонсор** приглашает по реф-ссылке; в ручном режиме выбирает слот в своём поддереве.
- **Админ (по роли):**
  - *Владелец* — всё: участники, настройки плана, роли.
  - *Финансы* — отчёты (выплаты вне цикла; роль заведена под Фазу 3).
  - *Лидер* — своя ветка (scope-ограничение видимости).
  - *Саппорт* — просмотр/помощь, без изменений плана.
  Управляет участниками (поиск/сегментация), настраивает проценты/пороги рангов/режим размещения.
- **Telegram:** бот онбордит и шлёт уведомления; Mini App открывает тот же кабинет внутри TG.

## 4. Функциональный объём (IN)

### 4.1 Реальная сеть и генеалогия
- Сущность **Member** (реальная), привязка к `CalculatorUser`.
- **Нормализованное** хранение дерева: Postgres `ltree`/closure — два графа: placement (бинар)
  и sponsorship (ЛП). **НЕ JSON-блоб.**
- Регистрация → **размещение** (`PlacementStrategy`: авто-спилловер слабая нога BFS / ручной
  слот; переключатель компании). **Конкурентная безопасность** (локи/версии) при размещении.
- Миграция расчёта: `CompensationEngine` строит `Network` **из БД**, считает на живых данных
  (golden-тесты ядра сохраняются зелёными).

### 4.2 Активация пакета (мок)
- Пакеты Bronze/Silver/Gold (PV из ядра). Партнёр выбирает пакет → «оплачено» (мок) →
  **идемпотентное событие активации** (idempotency-key) → движок пересчитывает объёмы → ранги → бонусы.
- Никакого ledger/кошелька/реальной оплаты — только событие и снимок результата расчёта.

### 4.3 Кабинет партнёра (web + Mini App)
- **Дашборд дохода:** разбивка по 5 типам бонусов + **логика расчёта** каждой выплаты.
- **Прогресс рангов:** текущий ранг, условия следующего, прогресс-бар.
- **Дерево команды:** react-d3-tree (переиспользуем из калькулятора) по живым данным; поиск/фильтр.
- **Активация пакета** (мок), история активаций.
- Профиль, **реф-ссылка** (+ Telegram deep-link), статус лёгкого KYC.

### 4.4 Админка + RBAC
- **RBAC:** роли владелец/финансы/лидер/саппорт + permission-гейты; назначение ролей.
- **Управление участниками:** список/поиск/сегментация (ранг/активность/ветка).
- **Настройка плана:** проценты/пороги рангов/режим размещения — `PlanConfig` в БД, редактируемый.
- Просмотр дерева/веток (лидер — только своя ветка).

### 4.5 Telegram (бот + Mini App)
- **Бот:** регистрация/онбординг, deep-link приглашения, уведомления (новый реферал,
  начислен бонус, достижение ранга). Отдельный воркер (без ingress).
- **Mini App:** UI кабинета внутри Telegram, **initData-валидация по HMAC-SHA256**.
- LLM/markdown в уведомлениях (если будет) → нормализация под Telegram-HTML (глоб. правило).

## 5. Вне scope этого цикла (OUT)

- **Ledger двойной записи, кошелёк, заявки на вывод, выплаты USDT** (Фаза 3).
- **Реальная оплата** (USDT/карта, шлюзы Cryptomus/PSP) — только мок-активация (Фаза 4).
- **Глобальный бонус** (периодический пул, период-клозинг) — отдельный цикл.
- Reconciliation (нет ledger), мультивалютность.
- Полноценный док-KYC/Sumsub, 2FA админов (лёгкий KYC — минимально: email/телефон/TG).
- Каталог/корзина/autoship, мобильные нативные приложения.

## 6. Ограничения / стек

- Backend: **Laravel 10** (модульный), **PostgreSQL** (`ltree`/closure для дерева).
  Локальный dev переводим на Postgres (ltree не работает на MySQL/SQLite) — обновить
  `DB_CONNECTION`, docker-postgres для dev. Сверить с облачным `izigo-pg-beta`.
- Frontend: **Next 14** (web-кабинет/админка + Telegram Mini App), antd, react-d3-tree.
- Переиспользуем: доменное **ядро** `Modules/Calculator/Domain`, локальную auth
  (`CalculatorUser`/`CalculatorUserToken`), пакеты/ранги, 6 локалей, react-d3-tree.
- Калькулятор-витрина остаётся как маркетинг/онбординг (расчёт переводим на общее ядро).
- Observability: Sentry (есть SDK), Azure-теги, секреты — Key Vault. Деплой — ACA.
- Telegram-бот: **@Izigopro_mlm_bot**; bot-token в Key Vault `izigo--beta--TELEGRAM-BOT-TOKEN`
  (`kv-bronxtc-dev`). Воркер тянет из KV (managed identity), не из env/файлов.

## 7. Критерии приёмки

1. Партнёр регистрируется по реф-ссылке (web/TG) → **ставится в дерево** (оба режима +
   переключатель работают) → **активирует пакет** (мок) → `CompensationEngine` считает
   все бонусы **на живых данных** → кабинет показывает **реальный доход** (разбивка +
   логика расчёта) и **дерево команды**.
2. Генеалогия **нормализована** (Postgres `ltree`), не JSON; размещение конкурентно-безопасно.
3. Витрина-калькулятор считает через **то же доменное ядро**; golden-тесты ядра зелёные.
4. Админ-RBAC: 4 роли с гейтами; управление участниками; настройка плана (проценты/пороги/режим)
   из админки применяется к расчёту.
5. Telegram: бот шлёт уведомления (реферал/бонус/ранг), Mini App открывает кабинет (initData валиден).
6. Тесты: генеалогия/размещение/идемпотентность активации/RBAC-гейты — зелёные; ядро не сломано.

## 8. Открытые вопросы на Гейт 2 (план)

- Точный способ хранения дерева: `ltree` vs closure table (или оба графа по-разному).
- Стратегия пересчёта: on-read снимок vs кэш-снимок результата активации (без ledger).
- Разбивка на под-этапы поставки (вертикальный срез → каналы → полнота админки).
- Где живёт `PlanConfig` (таблица + сидер) и как инъецируется в репозитории ядра.
