# ТЗ: 7-дневное лид-окно с изменяемым спонсором до первой покупки

**Дата:** 2026-06-23
**Тип:** изменение бизнес-логики регистрации/спонсорства (backend + Mini App). L-уровень.
**Конвейер:** `/armanda`. Этот файл — источник истины для Гейтов 2–4.

---

## 1. Цель

Ввести понятие **лида** и сделать спонсора **изменяемым до первой покупки**, фиксируя его
навсегда только после активации первого пакета.

Сейчас система делает обратное: на первом же заходе в Mini App создаётся `Member`, он сразу
ставится в бинарное дерево, `sponsor_id` фиксируется навсегда, повторная привязка/смена
невозможны, неактивные участники не удаляются. Это неверно для воронки «лид → клиент».

## 2. Целевое правило (от пользователя — дословно зафиксировано)

- Перешедший по реф-ссылке, но **не купивший** пакет = **ЛИД** (не клиент/партнёр).
  Закреплён за спонсором **только на 7 дней**.
- Пока лид не сделал первую покупку, спонсора можно менять **двумя** способами:
  - **(а)** переход по другой реф-ссылке в пределах окна → перепривязка **last-click-wins**;
  - **(б)** явная **кнопка в Mini App** «сменить спонсора».
- **Первая покупка** (активация любого пакета) **фиксирует спонсора навсегда** — после этого
  смена запрещена (deny-by-default).
- Через **7 дней без покупки** лид **открепляется и становится свободным**: следующий переход
  по любой рефке привязывает заново (можно к другому спонсору).

## 3. Решения по модели (подтверждены пользователем на Гейте 1)

1. **Лид — ВНЕ бинар-дерева до первой покупки.** Лид = мягкая привязка (запись «pending
   лид»: telegram_id + спонсор + срок), он **не занимает слот** в бинарном дереве. Спонсор
   видит лида как «pending». В бинар-дерево лид попадает **только при первой покупке** —
   обычным `PlacementService::place()` под уже-зафиксированного спонсора.
   - *Почему так:* в спилловер-дереве «позиция» ≠ «спонсорство». Если лид стоит в дереве, под
     него по BFS-спилловеру могут встать чужие оплатившие участники → лид перестаёт быть
     удаляемым/переносимым листом. Вынесение лида из дерева снимает конфликт и держит вход
     движка бонусов чистым (движок видит только оплативших).
2. **Лид НЕ рекрутирует до своей покупки.** Реф-ссылка/`ref_code` лиду **не выдаётся**, пока он
   сам не активировал пакет → у лида никогда нет даунлайна.
3. **Смена спонсора лида** = смена ссылки на спонсора в записи лида (одно поле), без какого-либо
   движения в дереве (лида в дереве нет).
4. **Истечение 7 дней** = открепление: запись лида удаляется/освобождается так, чтобы
   `telegram_id` мог заново привязаться под любым спонсором при следующем переходе.

## 3a. Доменное разграничение: личный реферал vs бинар (подтверждено пользователем)

Две независимые оси, которые сейчас в Mini App перепутаны:

- **Личный реферал = спонсорство (`sponsor_id`).** «Личный» определяется ТОЛЬКО тем, по чьей
  реф-ссылке человек зарегистрировался **и купил**. Личный реферал может стоять на любой
  глубине бинар-дерева (10-я, 1000-я) — глубина к «личности» отношения не имеет. Это прямое
  следствие модели лида: при первой покупке `sponsor_id` фиксируется = человек становится
  твоим личным рефералом.
- **Бинарное дерево = размещение (`parent_id`/`position`).** Всего 2 ветки. Люди сверху по
  спилловеру ставят в твою команду **своих** личных рефералов (чужих тебе по спонсорству) —
  они твоя «бинар-команда», дают тебе бинар-объём, но НЕ твои личные рефералы.

**Текущее состояние (баг, подтверждён кодом):**
- Счётчик «Личных»/«Приглашено» во вкладках «Команда» и «Профиль» = `tree.children.length`
  (`MiniAppShell.js:324`) — это прямые бинар-ноги, **максимум 2**. Помечено «личных», но это
  бинар, а не спонсорство.
- Настоящий счёт личных (по `sponsor_id`) есть только во вкладке «Ранг» (`CabinetService.php:81`)
  — и только числом.
- **Списка личных рефералов** (`sponsor_id = я`, любая глубина) нет: ни эндпоинта, ни хелпера.
- Бэк концептуально разделяет `sponsor_id` и `parent_id`/`position` корректно — наружу не выведено.

## 4. Пользователь и сценарии (user stories)

1. **Новый лид.** Аноним открывает Mini App по `?startapp=<ref_code>` → создаётся лид,
   привязанный к спонсору, со сроком +7 дней. В кабинете видит состояние «вы лид, активируйте
   пакет», спонсора и оставшийся срок. Реф-ссылки у него нет.
2. **Перепривязка по другой рефке.** Лид в пределах окна открывает Mini App по рефке другого
   спонсора → привязка меняется на нового спонсора (last-click-wins), срок (политику обновления
   срока зафиксировать в плане — см. §8).
3. **Явная смена спонсора.** Лид в кабинете жмёт «сменить спонсора», вводит/выбирает реф-код
   нового спонсора → привязка меняется. Валидируется: спонсор существует, не сам лид.
4. **Первая покупка = фиксация.** Лид оплачивает пакет → на подтверждённой оплате: лид
   промоутится в `Member`, ставится в бинар-дерево под текущего (замкнутого) спонсора,
   `status=active`, выдаётся `ref_code`/реф-ссылка, идёт обычный пересчёт. Спонсор зафиксирован.
5. **Запрет смены после покупки.** Активный участник пытается сменить спонсора (любым способом,
   включая переход по чужой рефке) → **отказ**, спонсор не меняется.
6. **Истечение.** Лид не купил за 7 дней → шедулер открепляет его. Следующий заход по рефке
   (того же или другого спонсора) создаёт нового лида заново.
7. **Уже активный заходит по чужой рефке.** Игнор реф-параметра, остаётся как есть.
8. **Личные рефералы корректно.** Активный участник видит верный счётчик личных рефералов
   (по `sponsor_id`, не ≤2) и список «Мои личные рефералы» — всех, кто зарегался+купил по его
   рефке, на любой глубине бинар-дерева, с пометкой их позиции/глубины в бинаре. Бинар-команда
   (вкладка «Команда», дерево) и личные рефералы визуально разведены по смыслу.

## 5. Границы (НЕ входит в эту итерацию)

- **Движок `Modules/Calculator` (расчёт бонусов) НЕ трогаем** — только его вход (когда узел
  попадает в дерево). Активация/пересчёт остаются как есть.
- Бот-команда `/сменить спонсора` (вариант (б) через бота) — **вне MVP** (у бота сейчас нет
  инфраструктуры команд кроме /start /app /help). Смена через кнопку в Mini App покрывает (б).
  Бот-команда — кандидат на следующую итерацию.
- Ручной перенос узлов в дереве (owner B2 `PlacementAdminService::move`) — не меняем.
- Менять валюту/способ оплаты/auth — нет (USD + Telegram, как есть).

## 6. Ограничения (стек/проект)

- Backend: Laravel, `Modules/Calculator`. Frontend: Next.js Mini App (`src/views/miniapp/`).
- Auth/идентичность: Telegram `initData`, identity = `telegram_id`.
- БД: PostgreSQL (ltree для дерева). Тесты на PostgreSQL (`izigo_test`), не SQLite.
- CI не мигрирует/не сидит — миграции в `docker/start.sh`.

## 7. Обязательные проверки (тесты пропорционально риску — auth/структура)

Негатив-кейсы (обязательны):
- смена спонсора **после** активации → запрещена (member остаётся со старым спонсором);
- self-referral (привязка к самому себе) → запрет;
- неизвестный/невалидный реф-код при смене → отказ, привязка не меняется;
- истёкшее окно: смена/действия лида после 7 дней → лид откреплён, операции корректны;
- повторная привязка по рефке в окне → last-click-wins, старая привязка снята.

Позитив/регресс:
- лид → покупка → корректная постановка в дерево под замкнутого спонсора, status=active,
  выдан ref_code;
- **личные рефералы:** эндпоинт возвращает всех `sponsor_id = me` (на разной глубине бинара),
  счётчик совпадает с числом записей; бинар-`team-tree` остаётся независимым;
- **golden-регресс движка** (`tests/Verification/`) — расчёт 4 бонусов не изменился
  (лиды вне дерева не влияют на вход движка).

## 8. Открытые вопросы для Гейта 2 (плановые, code-derivable — решаю в плане, не блокеры)

- Хранение лида: новая таблица `leads` vs. флаг на `members` (склоняюсь к отдельной `leads` —
  `members` с `parent_id=NULL` ломает индекс единственного корня).
- Точка «замка»: подтверждённая оплата (активация) vs. создание заказа. Склоняюсь к
  **подтверждённой оплате** (до неё лид изменяем; согласуется с TTL pending).
- Как `MiniAppAuth::resolveMember` различает три случая (активный member / существующий лид /
  новый) и что возвращает кабинету для лида.
- Промоушн лид→member на оплате: где встроить в `OrderService`/`PaymentService` →
  `place()` + `ActivationService::activate()` атомарно.
- Политика срока при перепривязке (сбрасывать 7 дней заново или сохранять исходный дедлайн).
- Ослабление текущего dedup-guard и `telegram_id unique` так, чтобы лид мог быть пересоздан
  после истечения, но активный member оставался уникальным.
- Шедулер истечения лидов (по аналогии с TTL pending в `PaymentService::pollPending`).
- Форма эндпоинта личных рефералов (`GET /cabinet/personal-referrals`?): список `sponsor_id=me`
  с полями (имя, статус, ранг, глубина/сторона в бинаре, дата). Чинить ли счётчик «Личных» в
  Team/Profile на этот же источник (вместо `tree.children.length`).
