# ТЗ: Фаза 4 — Commerce и платежи IziGo (модель A, товарная; TON/USDT)

**Дата:** 2026-06-21 (обновлено 2026-06-21: приём развёрнут с Wallet Pay на **TON Pay**).
**Тип:** крупная фича (backend + frontend + БД + платёжный шлюз + бот). Прогон через `/armanda`.
**Статус:** Гейт 2 — план. Backend S1–S8 реализован под Wallet Pay; идёт разворот приёма на TON Pay.

> **Δ 2026-06-21 — разворот приёма на TON Pay (non-custodial).** Wallet Pay (custodial) заменён на
> **TON Pay** (`@ton-pay/*`, поверх TON Connect): деньги идут **напрямую на наш TON-кошелёк**, без
> процессора. Следствия: (1) бэкенд ОБЯЗАН сам валидировать платёж on-chain (адрес+сумма+memo=payment_id);
> (2) подтверждение — не webhook процессора, а **poll TON-сети** (паттерн как у on-chain выплат);
> (3) SDK `@ton-pay/api` — Node, поэтому на фронте TON Pay UI, а валидация — в нашем Laravel-бэкенде.
> Прежнее «TON Connect исключаем» — ОТМЕНЕНО (TON Pay и есть TON Connect). `WalletPayGateway` остаётся
> запасным драйвером за абстракцией `PaymentGateway`.
**Связано:** `roadmap.md` (Фаза 4), `docs/specs/2026-06-21-phase3-financial-core.md` (ledger/wallet),
память `izigo-phase2-progress`, `izigo-prod-map`. Концепт NEXUS (`docs/Binar_Company_2027.docx`) —
частично подтягивается этим решением (on-chain/USDT), но как документ остаётся отдельным.

## 1. Цель

Дать партнёру **реальную покупку товара за деньги**, которая порождает PV/оборот, прогоняется через
домен-ядро Фаз 1–3 (активация → бонусы → ledger → кошелёк) и закрывает обе стороны денег —
**приём оплаты** и **выплату** заработка — в **одном крипто-рельсе (USDT в сети TON)**.

Exit (из roadmap): **заказ → PV/объём → комиссия → кошелёк, end-to-end**, плюс автоматические
**on-chain выплаты в USDT**.

## 2. Решения Гейта 1 (подтверждены пользователем)

1. **Объём — полная Фаза 4** (каталог, корзина, checkout, autoship/подписки, retry неудачных
   платежей, трекинг исполнения, связка autoship↔квалификация, приём+выплаты). Собираем **поэтапными
   инкрементами** внутри фазы (см. §7), но цель итерации — весь объём.
2. **Модель бизнеса — A (товарная):** реальный каталог физических товаров с ценой и PV.
3. **Платёжный рельс — крипто TON/USDT** (разворот с фиатного Telegram Payments).
   - **Учётная единица — USDT** (стейблкоин Tether в сети TON). Ledger уже в целых «центах» →
     трактуем как USDT-центы; мультивалютная конверсия НЕ входит.
   - **Приём — TON Pay** (non-custodial, `@ton-pay/*` поверх TON Connect). Деньги идут **напрямую
     на наш merchant TON-кошелёк**; процессора-держателя нет. На фронте — TON Pay UI (`@ton-pay/ui-react`
     + `@tonconnect/ui-react`), на бэкенде (Laravel) — **самостоятельная валидация on-chain** платежа
     по `адрес + сумма + memo (= payment_id/external_ref)`. **Товар/активацию выдаём только после
     подтверждения транзакции в сети** (статус `pending → confirmed`).
   - **Подтверждение — poll TON-сети** (toncenter/TON API) командой, не webhook процессора. Memo =
     наш `external_ref` (`pay:{id}`).
   - **Wallet Pay** остаётся реализованным запасным драйвером за абстракцией `PaymentGateway` (не
     удаляем), но активный приём — TON Pay.
   - **Крипто-приём заменяет фиат:** Telegram Payments отменён.
4. **Выплаты — on-chain в USDT** (TON), автоматически из единого фин-интерфейса. Заменяет ручную
   выплату Фазы 3 (заявка/одобрение остаются, «выплачено вручную» → «отправлено on-chain»).
5. **KYC — двухслойно:**
   - **Intake (эта фаза):** Telegram Passport как сбор документов + **ручной аппрув** комплаенс/
     финансистом в админке + **пороговая логика** (мелкие суммы без KYC; перед первой выплатой /
     выше порога — требуем Passport + аппрув).
   - **Верификация/AML (Фаза 5):** полноценный движок (Sumsub/аналог) — подлинность, liveness,
     санкции/PEP. Passport остаётся UX-входом. **Telegram Passport сам KYC НЕ делает** (только
     зашифрованный сбор документов) — это зафиксировано как ограничение.

## 3. Пользователь и сценарии (user stories)

- **US-1 (каталог):** Партнёр/клиент видит каталог товаров с ценой (USDT) и PV. Может открыть карточку.
- **US-2 (корзина и checkout):** Складывает товары в корзину, оформляет заказ, видит итог в USDT и
  суммарный PV.
- **US-3 (оплата TON Pay):** Оплачивает заказ через TON Pay (TON Connect): подключает кошелёк
  (Tonkeeper/@wallet/…), подписывает перевод USDT с memo=`payment_id` на наш адрес. Пока tx в сети
  не подтверждена — UI «подтверждаем оплату»; бэкенд опросом сети валидирует адрес+сумму+memo и лишь
  тогда переводит заказ в `paid`. (Опц. on-ramp: купить TON картой прямо в чекауте.)
- **US-4 (заказ → начисление):** Оплаченный заказ создаёт `ActivationEvent` (PV/пакет) → домен-ядро
  считает бонусы → дельта-проводки в ledger → баланс спонсоров в кошельке растёт (как в Фазе 3).
- **US-5 (трекинг исполнения):** Партнёр видит статус заказа: `paid → processing → shipped →
  delivered` (+ `cancelled/refunded`). Админ меняет статус, может указать трек-номер (текст).
- **US-6 (autoship/подписка):** Партнёр включает автозаказ (периодичность). К сроку списания система
  готовит платёж; **в крипте нет авто-дебета карты** (TON Pay подписок нативно НЕ поддерживает) →
  списание идёт **с внутреннего USDT-баланса кошелька** (партнёр пополняет его через TON Pay top-up),
  иначе — напоминание оплатить вручную.
  Успешный autoship-заказ так же даёт PV и поддерживает квалификацию.
- **US-7 (retry неудачного autoship):** Если на дату списания средств/оплаты нет — система делает
  повторные попытки/напоминания по расписанию **д.3 / д.7 / д.14**; после исчерпания — autoship
  ставится на паузу, квалификация не продлевается.
- **US-8 (вывод on-chain):** Партнёр запрашивает вывод ≤ доступного баланса, указывает свой
  **TON-адрес для USDT**. Финансист одобряет → средства в холд → система **отправляет USDT on-chain**
  → заявка `paid` с tx-hash (виден партнёру для прозрачности).
- **US-9 (KYC-гейт):** Перед первой выплатой (или выше порога) партнёр проходит Passport-intake;
  финансист аппрувит в админке; без аппрува выплата заблокирована.
- **US-10 (админ/каталог):** Админ управляет каталогом (товар: название, описание, цена USDT, PV, SKU,
  активность, остаток — опц.), видит заказы, autoship-подписки, заявки на вывод, KYC-очередь.

## 4. Границы (НЕ входит в эту итерацию)

- Любой **фиатный** приём (Telegram Payments, карты, банки) и мультивалютная конверсия. *(Примечание:
  TON Connect теперь ВХОДИТ — через TON Pay; прежнее исключение снято разворотом приёма.)*
- Нативные крипто-**подписки** (TON Pay их не поддерживает) — autoship реализуем списанием с
  внутреннего баланса, а не авто-дебетом кошелька.
- Полноценный **AML/санкционный движок** (Sumsub и пр.) — Фаза 5; здесь только Passport-intake +
  ручной аппрув.
- Период-клозинг (`commission run`) — остаётся хвостом Фазы 3.
- Интеграция с перевозчиками/складом (реальный трекинг доставки, ERP) — только статусы + трек-номер
  текстом.
- Промокоды/скидки/налоги/возвраты частями — базовый refund (полный) допустим, сложные сценарии нет.

## 5. Доменные сущности (высокоуровнево; детали схемы — Гейт 2)

- **Product** — товар каталога: name, description, price_usdt (центы), pv, sku, is_active, stock?(опц).
- **Cart / CartItem** — корзина партнёра (или клиента) до оформления.
- **Order / OrderItem** — заказ: member_id, items, total_usdt, total_pv, status
  (`pending_payment → paid → processing → shipped → delivered`, ветки `cancelled/refunded`),
  shipping_info (текст), tracking_no?(текст).
- **Payment** — платёж по заказу/пополнению: provider=`ton_pay` (или `wallet_pay`-fallback),
  external_ref (=memo `pay:{id}`), external_id/reference, amount_usdt, status, raw payload, idempotency.
  **Idempotent** по external_ref; для TON Pay подтверждение — валидацией on-chain (адрес+сумма+memo).
- **AutoshipSubscription** — подписка: member_id, items/preset, interval, next_charge_at, status
  (`active/paused/cancelled`), retry-состояние (счётчик/следующая попытка д.3/7/14).
- **PayoutTransaction** — on-chain выплата: withdrawal_request_id, to_address, amount_usdt, tx_hash,
  status (`queued → broadcast → confirmed/failed`).
- **KycRecord** — статус KYC партнёра: level/threshold, source=`telegram_passport`, документы(ссылки/
  зашифровано), review_status (`pending/approved/rejected`), reviewer, дата.

Связь с Фазами 1–3: оплаченный Order **порождает `ActivationEvent`** (существующий триггер начислений)
→ `LedgerService`/`WalletService` без изменения их контракта. Внутренний USDT-баланс кошелька =
существующий `member_available` (трактуем как USDT). Заявки на вывод — существующий
`WithdrawalService`, расширяем шагом on-chain отправки.

## 6. Интеграции и секреты (Azure Key Vault — обязательно)

- **TON Pay (приём, non-custodial):** наш **merchant TON-адрес** получателя → KV
  (`izigo--<env>--TON-MERCHANT-ADDRESS`); ключ доступа к TON API (toncenter) для опроса сети →
  `izigo--<env>--TON-API-KEY`. Деньги идут напрямую на адрес — приватный ключ приёма серверу НЕ нужен.
  Подтверждение — **валидацией on-chain** (адрес+сумма+memo=`external_ref`), товар/активация только
  после `confirmed`. Фронт: TON Connect manifest по абсолютному URL. Зависимости (пин): `@ton-pay/api`,
  `@ton-pay/ui-react` (v0.3.x, beta — пинить!), `@tonconnect/ui-react`.
- **Wallet Pay (fallback-драйвер):** api-key/webhook-secret → KV (`izigo--<env>--WALLETPAY-*`). Не
  активен, но реализован за абстракцией.
- **On-chain выплаты (USDT/TON):** ключ кошелька-плательщика — **критичный секрет**. Приватный ключ
  отправляющего кошелька хранить ТОЛЬКО в Key Vault (в идеале — отдельный hot-wallet с лимитом);
  никогда в env/коде. Подпись/отправка транзакции — через серверный сервис, не на клиенте.
- **Telegram Passport:** приватный ключ расшифровки документов → KV. Документы в БД хранить
  зашифрованными/как ссылки, не в открытом виде.
- **Sentry** — расширить трейсинг на новые потоки (checkout, webhook, payout); ошибки приёма/выплат
  не должны ронять основной поток (best-effort на «бонусных» уведомлениях).
- LLM-текст наружу (если будет в уведомлениях бота) — через нормализацию `md → Telegram-HTML`.

## 7. Инкременты внутри фазы (порядок сборки; детальный план — Гейт 2)

1. **Каталог + корзина + заказ** (без оплаты): Product/Cart/Order, админ-каталог, UI каталога/корзины.
2. **Приём Wallet Pay:** Payment, invoice/checkout, webhook (идемпотентный, подпись), Order→`paid`.
3. **Заказ → начисление:** Order(`paid`) → `ActivationEvent` → ledger/wallet (переиспользуем Фазу 3).
4. **Трекинг исполнения:** статусы заказа + трек-номер, UI партнёра и админки.
5. **Autoship + retry:** подписки, списание с внутреннего USDT-баланса, расписание д.3/7/14, связка с
   квалификацией; планировщик (Laravel scheduler/queue).
6. **On-chain выплаты:** PayoutTransaction, отправка USDT, tx-hash в заявке; единый фин-интерфейс.
7. **KYC-intake:** Telegram Passport сбор + админ-аппрув + пороговый гейт перед выплатой.

## 8. Открытые вопросы для Гейта 2 (план)

- **Autoship в крипте:** подтвердить модель «списание с внутреннего USDT-баланса + пополнение через
  Wallet Pay» (рекомендация) vs только напоминания об оплате. Рекомендация — внутренний баланс
  (переиспользует e-wallet Фазы 3, даёт настоящий авто-дебет).
- **Хранение средств:** где лежит USDT между приёмом и выплатой — единый hot-wallet проекта; лимиты/
  алёрты на баланс; политика пополнения cold→hot.
- **Пороги KYC** (суммы/триггеры) и какие документы требуем в Passport.
- **Refund-политика** (полный возврат: откат PV/начислений = clawback-проводки Фазы 3?).
- **Точность on-chain:** комиссии сети TON, минимальные суммы вывода, обработка `failed` транзакций.
- Региональная доступность **Wallet Pay** под целевой рынок (проверить перед интеграцией).

## 9. Тесты (пишем из этого ТЗ, до реализации)

- Заказ→оплата(webhook идемпотентный)→`ActivationEvent`→дельта в ledger→баланс кошелька (e2e US-4).
- Двойной/повторный webhook не задваивает оплату и начисления (идемпотентность).
- Autoship: успешное списание с баланса даёт PV; недостаток средств → retry д.3/7/14 → пауза (US-6/7).
- Вывод on-chain: одобрение → холд → отправка → `paid` с tx-hash; `failed` → возврат холда (US-8).
- KYC-гейт: без аппрува выплата заблокирована; после аппрува проходит (US-9).
- Refund заказа откатывает PV/начисления (clawback) — если refund в scope.

## 10. Definition of Done

- Партнёр оформляет и оплачивает заказ через Wallet Pay; начисления видны в кошельке (e2e).
- Autoship с retry-расписанием работает и поддерживает квалификацию.
- Вывод заработка приходит партнёру **on-chain в USDT** с видимым tx-hash.
- KYC-intake + пороговый гейт перед выплатой работает.
- Тесты §9 зелёные; lint/format пройдены; Sentry покрывает новые потоки; секреты только в KV.
