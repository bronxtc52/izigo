# ТЗ: Закрытие P1-находок продакшн-ревью (production hardening)

**Дата:** 2026-07-02
**Статус:** Гейт 1
**Источник:** продакшн-ревью 2026-07-02 (6 параллельных ревью: security, деньги, Блок C+AI, фронтенд, ops, тесты). Все 4 денежных P1 подтверждены построчным чтением кода.
**Скоуп:** 11 P1 + 3 дешёвых P2 (решение Гейта 0). Остальные P2 — отдельной итерацией.

---

## Цель

Довести платформу от «бета под присмотром» до безнадзорного продакшена: устранить все сценарии
потери/задвоения денег, сделать rollout-гейт фиче-флагов реальным, закрыть слепые зоны
наблюдаемости и дыры деплой-конвейера.

## Форма сдачи (DoD, согласовано на Гейте 0)

- **3 тематических PR:** PR-1 backend, PR-2 frontend, PR-3 ops. Ссылки в финальном отчёте.
- Полный тестовый сьют зелёный (350+ существующих + новые тесты на каждый денежный фикс).
- Мердж в main (= прод-деплой) — **только после одного финального «ок»** пользователя,
  порядок: backend → ops → frontend.
- Sentry-проект фронта создаю сам через API self-hosted Sentry (токен из KV),
  DSN → KV `izigo--prod--SENTRY-DSN-FRONTEND`.

## PR-1 — Backend (деньги + флаги + рассылка)

### B1. Poison-платёж не должен останавливать конвейер подтверждений
`PaymentService::pollPending` (строки ~202-212): исключение из `confirmPayment()` одного платежа
валит весь цикл и TTL-блок. **Фикс:** try/catch вокруг тела итерации, лог + Sentry per-payment,
цикл продолжается. TTL-блок выполняется всегда.
*Сценарий из ТЗ:* платёж с осиротевшим lead_id больше не блокирует подтверждение остальных.

### B2. Сериализация активаций против двойного начисления
`ActivationService::recompute` (~128-194): полный delete/rewrite снимка earnings + дельта-проводки
без глобальной сериализации → конкурентные активации (тик поллера + ручной `payments/{id}/check`)
задваивают начисления. **Фикс:** `pg_advisory_xact_lock` с фиксированной константой в начале
транзакции `activate()` — все пересчёты сети строго последовательны.

### B3. Autoship: списание и активация атомарны
`AutoshipService` (~139-153): `ledger->charge` коммитится отдельно, `create+markPaid` вне
транзакции → сбой = деньги списаны, активация потеряна навсегда. **Фикс:** обернуть
charge + create + markPaid в одну `DB::transaction` (по образцу `PaymentService::applyPaid`);
poison-подписка не прерывает прогон остальных (try/catch per-sub — заодно закрывает P2-7).

### B4. TTL-экспирация не съедает реально оплаченные платежи
`PaymentService::pollPending` (~219-225): **Фикс:**
- не экспирировать платёж, если его последний опрос завершился ошибкой сети/API
  (`TonPayGateway::pollStatus` различает «опросили, перевода нет» и «опрос не удался»);
- warning в лог/Sentry при экспирации платежа;
- админ-ручка `POST /admin/payments/{id}/recheck` (owner/finance) — принудительный ре-опрос
  expired/pending платежа с подтверждением при найденных деньгах.

### B5. Enforcement фиче-флагов c1..c7 на бэкенде
Контроллеры Блока C (`Notification`, `Ticket`, `Copartner`, `MemberExport`, `Monitoring`,
`Translation`) не проверяют свои флаги — гейт «фичи OFF» иллюзорен. **Фикс:** middleware
`feature.flag:{alias}` на группы роутов (по образцу проверки в `AiAssistantController`),
403 при выключенном флаге. `FeatureFlagController` (c3) — owner-only, без флага (by design).

### B6. Рассылка: bulk-постановка + идемпотентность повтора
`BroadcastService`/`NotificationService`: O(3N) запросов синхронно + повтор после таймаута
создаёт дубли. **Фикс:** bulk-insert чанками (500 строк на insert); dedup-ключ outbox не
зависит от id новой broadcast-записи (детерминирован по содержимому+получателю), повторный
запуск «застрявшей» processing-рассылки допоставляет только недостающих.

### B7. Валидация TON-адреса на бэкенде
`WithdrawalService::create`: бэк не проверяет формат `payout_details`. **Фикс:** серверная
валидация user-friendly TON-адреса с CRC16-чексуммой, отклонение testnet-флага → 422.
Без новых зависимостей (чексумма — ~20 строк, алгоритм известен) либо lightweight-порт;
выбор зафиксировать в плане Гейта 2.

**Тесты PR-1 (обязательны, деньги):** poison-платёж не блокирует соседей; конкурентные активации
не задваивают ledger (advisory lock); autoship-сбой откатывает списание; экспирация при ошибке
опроса не происходит; recheck-ручка подтверждает expired-платёж; 403 по каждому флагу c1..c7;
повтор рассылки не дублирует доставленных; невалидный/testnet TON-адрес → 422.

## PR-2 — Frontend

### F1. Sentry для Next.js
`@sentry/nextjs` (client + server), DSN из `NEXT_PUBLIC_SENTRY_DSN` (build-arg, публичный
по природе), release = `GITHUB_SHA`, sourcemaps на self-hosted best-effort (SENTRY_URL
обязателен, org=sentry). `send_default_pii=false`, initData в события не попадает.

### F2. Error boundaries
`src/app/error.js` + `global-error.js`: сообщение + кнопка «перезагрузить», ошибка уходит
в Sentry. Стиль Aurora.

### F3. TON-адрес с чексуммой на фронте
`isTonAddress` в `MiniAppShell.js:35` → `Address.parseFriendly()` из уже имеющейся
`@ton/core` в try/catch, отклонять testnet.

### F4 (P2). Guard от prototype pollution в i18n
`src/common/i18n.js:56-69`: скипать сегменты `__proto__|constructor|prototype`
при разворачивании dot-ключей оверрайдов.

### F5 (P2). Анти-двойная оплата после таймаута поллинга
`TonPayCheckout.js`: после успешного `sendTonPayment` и исчерпания MAX_POLLS не возвращать
кнопку «Оплатить кошельком» — оставлять «Проверить оплату» + подсказку «перевод уже отправлен».

## PR-3 — Ops

### O1. Миграции не проглатываются
`docker/start.sh:9`: убрать `|| true` у `php artisan migrate --force` (упавшая миграция =
упавший старт контейнера, ACA не переключит трафик). У `config:cache`/сидеров — оценить
построчно в Гейте 2 (сидер идемпотентный — вероятно тоже без `|| true`).

### O2. Тесты в CI перед деплоем
`.github/workflows/deploy.yml`: job `test` (services: postgres:16, БД izigo_test, `php artisan
test`) — деплой-jobs зависят от него. Фронт: `npm run lint && npm run build` в том же гейте.

### O3 (P2). `bot.catch()` в grammY-боте
`mh-calc-bot/src/bot.js`: глобальный обработчик — лог + Sentry, long-polling не падает.

**Границы PR-3:** пайплайн деплоя не перестраиваем (тот же acr build/containerapp update),
бот в CI не добавляем (отдельная итерация), liveness-probes ACA — вне скоупа.

## Границы (что НЕ входит)

- Движок расчёта бонусов (Modules/Calculator ядро CompensationEngine) — не трогаем;
  advisory-lock в B2 — обвязка вокруг вызова, не сам движок.
- Остальные P2: liveness-probes, C5-маскирование в admin/members, окно матчинга 100 переводов,
  гонка лид-экспирации (`created`-статус), logout Sanctum, max_age initData, Next.js CVE-апгрейд,
  webvisor Метрики, `.env.example`, retention бэкапов — **следующая итерация**.
- On-chain выплаты — вне скоупа by design (выплаты вручную, решение 2026-06-23).
- Ребрендинг статусов ассистента (200 → 503) — не трогаем (закреплено тестом+фронтом).

## Критерии приёмки

1. Все 14 пунктов (B1-B7, F1-F5, O1-O3) реализованы и покрыты тестами, где применимо.
2. `php artisan test` зелёный целиком; `next build` зелёный; тесты бота зелёные.
3. Три PR открыты, в описании каждого — маппинг «фикс → находка ревью».
4. CI job `test` реально гоняется и падает при красных тестах (проверить на ветке).
5. После мержа (по «ок»): ревизии Healthy, smoke: оплата фейковым чек-флоу не регрессировала,
   403 на выключенный флаг Блока C, Sentry фронта ловит тестовое событие.
