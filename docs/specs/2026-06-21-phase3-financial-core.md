# ТЗ: Фаза 3 — Финансовое ядро IziGo (ledger + e-wallet + выводы)

**Дата:** 2026-06-21
**Тип:** крупная фича (backend + frontend + БД). Прогон через `/armanda`.
**Статус:** Гейт 1 — ТЗ (на подтверждении).
**Связано:** `roadmap.md` (Фаза 3), `docs/specs/2026-06-21-telegram-only-auth.md`, память `izigo-phase2-progress`.

## 1. Цель

Дать партнёру **реальный учёт заработанных денег и возможность их вывести**, а финансисту —
инструмент одобрения выплат. Превратить текущий destructive-снимок начислений
(`member_bonus_lines`/`member_earnings`, который при каждой активации удаляется и пересчитывается
с нуля) в опору для **иммутабельного финансового журнала (ledger двойной записи)**, из которого
формируется баланс кошелька и из которого происходит списание при выводе.

Снимок остаётся **витриной** (как зарабатывается доход по типам бонусов), ledger становится
**источником истины для денег**.

## 2. Решения Гейта 1 (подтверждены пользователем)

1. **Модель начисления — реалтайм при активации.** Каждая активация сразу пишет иммутабельные
   ledger-проводки по **дельте** начислений; баланс кошелька растёт мгновенно. Периодический
   `commission_run` (период-клозинг) в эту итерацию НЕ входит.
2. **Scope MVP — ядро:** ledger двойной записи + e-wallet (баланс/история) + заявки на вывод +
   approval финансистом. БЕЗ commission run, БЕЗ мультивалютной конверсии.
3. **Вывод — ручная выплата вне системы.** Партнёр указывает реквизиты текстом; финансист
   одобряет (списание из ledger + холд), платит вручную (банк/крипто) и отмечает «выплачено».
   Реальные платёжные шлюзы — Фаза 4. Жизненный цикл заявки:
   `requested → approved → paid` или `requested → rejected` (а также `approved → cancelled` при
   возврате средств, если выплата не состоялась).

## 3. Пользователь и сценарии (user stories)

- **US-1 (накопление дохода):** При активации пакета подо мной/в моей сети мой доход растёт.
  В ledger появляются проводки (referral/binary/leader/rank), баланс кошелька увеличивается на
  дельту. Я вижу это в кабинете как историю операций.
- **US-2 (баланс и история):** В кабинете (Mini App) вижу **доступный баланс**, **в холде**
  (по заявкам на выводе) и **историю операций** (начисления + выводы) с типом, суммой, датой.
- **US-3 (заявка на вывод):** Запрашиваю вывод на сумму ≤ доступного баланса, указываю реквизиты
  (текст). Сумма уходит в холд (доступный баланс уменьшается). Вижу статус заявки.
- **US-4 (одобрение, финансист):** Роль `finance`/`owner` в админке видит очередь заявок,
  открывает заявку (партнёр, сумма, реквизиты, текущий баланс), **одобряет** (фиксирует списание)
  или **отклоняет** (возврат суммы из холда в доступный баланс). После ручной выплаты отмечает
  заявку `paid`.
- **US-5 (аудит):** Любая денежная операция отражена в ledger иммутабельно (append-only),
  с привязкой к источнику (activation_event / payout) и идемпотентным ключом. Историю нельзя
  «переписать» — только компенсирующими проводками.

## 4. Ограничения

- Стек как есть: Laravel (`mh-calc-backend-main`, модуль `Modules/Calculator`) + Next App Router
  (`mh-calc-frontend-main`) + Postgres (ltree). Telegram-only идентичность (`Member.telegram_id`),
  доступ по initData; админка под `RoleMiddleware` (`owner/finance/leader/support`).
- **Деньги — целые центы в домене** (`Domain/ValueObject/Money`), `decimal(20,2)` в БД; конверсия
  через существующий паттерн (`centsToDecimal` / `Money::fromCents`). Float не вводить.
- **Идемпотентность обязательна** — переиспользуем паттерн `activation_events.idempotency_key`
  (unique) + `lockForUpdate`. Ledger-проводки от одной активации не должны задваиваться.
- Валюта одна (de-facto, конвертации нет). Балансы в одной валюте; мультивалюта — вне итерации.
- Тесты обязательны (PostgreSQL, БД `izigo_test`): двойная запись, идемпотентность, корректность
  баланса, холд/возврат, RBAC выводов, отсутствие отрицательного доступного баланса.
- Sentry уже подключён (backend) — новые финансовые ошибки попадают в трекинг.

## 5. Что входит в итерацию

- **Ledger двойной записи** (иммутабельный append-only журнал): проводки с источником и
  идемпотентным ключом; баланс = свёртка проводок (с денормализованным кэшем баланса для скорости).
- **Интеграция с активацией:** при `recompute` дельта начислений каждого затронутого узла пишется
  в ledger как корректирующие проводки (см. §7 — ключевая сложность).
- **E-wallet:** доступный баланс, баланс в холде, история операций — API + UI в кабинете (Mini App).
- **Заявки на вывод:** создание партнёром (сумма + реквизиты), холд средств; API + UI.
- **Approval-флоу для финансиста:** очередь заявок, одобрить/отклонить/отметить paid; API + UI
  в админ-разделе Mini App под `calculator.role:owner,finance`.
- Тесты на все сценарии §3.

## 6. Что НЕ входит (границы)

- **commission_run / период-клозинг** (выбрана реалтайм-модель).
- **Мультивалютная конверсия** (балансы в одной валюте).
- **Платёжные шлюзы / авто-выплаты** (фиат/крипто) — Фаза 4.
- **KYC / 2FA / комплаенс** — Фаза 5.
- **Commerce/заказы/autoship** — Фаза 4 (модель бизнеса A/B пока не выбрана; на Фазу 3 не влияет).
- Изменение комп-плана/формул начислений — остаются как есть (Domain-ядро не трогаем по сути).

## 7. Ключевая архитектурная сложность (для Гейта 2)

Текущая активация делает **глобальный destructive recompute**: `MemberBonusLine::delete()` +
`MemberEarning::delete()` + полная перезапись по всей сети. При активации у нижнего узла суммы
**вышестоящих** узлов тоже меняются (binary carryover, rank-compression, leader bonus-on-bonus).
Поэтому «дельта при активации» — это набор изменений по **многим** узлам, а не только по одному.

Ledger обязан остаться append-only. Значит при каждой активации:
1. считаем новый снимок начислений по сети (как сейчас);
2. сравниваем с предыдущим снимком по каждому узлу → **дельта на узел**;
3. пишем в ledger **корректирующие проводки** на дельту (положительные и отрицательные);
4. обновляем кэш баланса кошелька на дельту.

**Открытые вопросы Гейта 2 (решить при планировании):**
- **Clawback:** если recompute уменьшил доход узла ниже уже выведенной/захолженной суммы —
  как поступаем (запрет ухода баланса в минус, отрицательный «долговой» баланс, блок вывода)?
- Гранулярность проводок: одна агрегированная дельта на узел за активацию, или по типам бонусов.
- Схема счётов двойной записи: какие счета (earnings-источник ↔ wallet ↔ payout-холд ↔ выплачено).
- Привязка идемпотентности: ключ на (activation_event × узел) для проводок начисления.
- Денормализованный кэш баланса vs пересчёт на лету (производительность кабинета).

## 8. Критерии приёмки

- Активация → корректные иммутабельные проводки в ledger; повторная активация (тот же
  idempotency_key) НЕ задваивает проводки.
- Баланс кошелька = сумма проводок ledger (сходится «до цента»); кэш совпадает с пересчётом.
- Заявка на вывод ≤ доступного баланса; средства уходят в холд; нельзя вывести больше доступного.
- finance/owner одобряет/отклоняет; отклонение возвращает холд; `paid` фиксирует выплату.
- Партнёр без роли НЕ видит чужие выводы и админ-очередь (RBAC).
- Все денежные суммы — без float-потерь; тесты на PostgreSQL зелёные.
