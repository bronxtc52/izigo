# Гейт A — поправки к планам по итогам ревью совета моделей

**Дата:** 2026-07-12 · **Ревью:** docs/reviews/2026-07-12-mh-full-plan-gateA-review.md (request_changes, 11 must-fix)
**Статус:** поправки обязательны для всех кодинг-агентов волн; при конфликте с текстом плана задачи — приоритет у этого документа.

## Закрытие must-fix

**MF-1/MF-2 — порядок 60%-пула и формула (critical).**
Канон (единственный источник — T11 `PoolFactorService`):
- Числитель калибровки месяца `total_after_caps_cents` = структурная премия после индивидуальных
  капов + месячное накопление глобального бонуса.
  **НЕ входят в числитель:** лидерский бонус (DEC-029 — считается ОТ фактически выплаченного ПОСЛЕ
  калибровки, включение создало бы цикл); **реферальная премия** (решение владельца 2026-07-12,
  MF-W3-3 — платится мгновенно на ОС по спеке CAL-REF-001, clawback запрещён MF-4, поэтому вне
  60%-пула; обоснование: реферальная = фиксированный % от реальной покупки, не раздувает
  payout-vs-turnover); квал-награды (owner-approved исключение из DEC-014).
  Таким образом 60%-калибровке подлежат только структурная, лидерский (через базу) и глобальный.
- Целочисленная формула (центы/bps, без float):
  `pool_cap_cents = intdiv(base_bv_cents * 6000, 10000)`;
  `factor_bps = total_after_caps_cents == 0 ? 10000 : min(10000, intdiv(pool_cap_cents * 10000, total_after_caps_cents))`;
  применение: `paid_cents = intdiv(raw_cents * factor_bps, 10000)`, остатки распределения — по
  наибольшим дробям, сумма выплат строго ≤ pool_cap_cents.
- T08 читает **закоммиченный** factor_bps периода из `v2_pool_calibrations` и не вычисляет ничего сам.

**MF-4 — НС→ОС и месячная калибровка (принят вариант A совета).**
Структурная премия остаётся на НС до закрытия МЕСЯЦА и коммита калибровки; перевод НС→ОС
выполняется job'ом 1-го числа за оба полумесяца прошедшего месяца, уже умноженный на factor_bps.
Провизорных переводов 16-го и clawback-корректировок калибровки НЕТ (clawback-механика остаётся
только у возвратов T12). Отклонение от буквального «перевод 1/16» спеки зафиксировано (спека сама
оставила стык BLOCKER: DEC-019 / OPEN-POOL-02); партнёру НС виден как «начислено, ожидает
закрытия месяца».

**MF-5 — контракт резолвера политики.** Единственная сигнатура:
`PolicyVersionResolver::forDate(\DateTimeInterface $at): PolicyV2` (+ шорткат `current()`).
Возврат — доменный объект PolicyV2 (не array, не int). Владелец — T01, все прочие задачи
потребляют только этот интерфейс.

**MF-6 — один владелец переводов НС→ОС.** Команда и расписание — ТОЛЬКО T04
(`calc-v2:ns-os-transfer`, ежедневно, гейт «месяц закрыт и откалиброван», см. MF-4).
T02 отдаёт только сервис-операцию `NsToOsTransfer::executeForCalibratedMonth(...)`.
Команду `mh2:ns-transfer` и schedule-строку из плана T02 — убрать.

**MF-7 — grace CLIENT.** Канон: state остаётся `client`, отдельная колонка
`grace_expires_at` (nullable). Сканер: `state='client' AND grace_expires_at < now()`.
Отдельного state `client_grace` не существует; терминальный — `grace_expired`.

**MF-8 — единый словарь схемы V2 (обязателен для всех задач).**
Все таблицы V2 — с префиксом `v2_`:
`v2_policy_versions` (статусы: draft / active / retired — статуса APPROVED нет; активация
one-step owner-activate), `v2_wallet_lots`, `v2_calc_periods`, `v2_pv_lots`
(сразу с колонкой `reversal_of_lot_id` — закладывает T03, использует T12),
`v2_award_entitlements` (статусы: granted / on_hold / paid_out / forfeited),
`v2_pool_calibrations`, `v2_partner_states`.
T13 (админка) потребляет read/action-интерфейсы задач-владельцев и НЕ вводит собственных имён.

**MF-9 — награды не сгорают.** `LedgerV2::credit(..., ?DateTimeInterface $expiresAt)` — nullable;
`expireLots` пропускает `expires_at IS NULL`; T10 кредитует award-лоты с `null`.
Решение владельца в дефолтах Гейта A («award-лоты БС без сгорания») — подтверждено.

**MF-10 — rollback-плейбук cutover (T15).** Простой rollback (выключить флаг) допустим только
в окне «после включения V2 не оплачен ни один заказ». Иначе: компенсирующие V2-reversals по всем
заказам окна + V1-recompute backfill, всё под ACTIVATION_LOCK; шаги — в runbook T15, прогон
репетиции на staging-копии данных обязателен до PROD-гейта.

**MF-11 — текст политики v3 (T16).** Раздел лидерского бонуса описывает ТОЧНУЮ семантику
DEC-030: «блок без передачи» (при разнице статусов ≥ порога участник и его поддерево не порождают
лидерскую выплату получателю; бонус НЕ передаётся выше), формулировку «компрессия минус два» с
передачей — не использовать.

## Принятые nice-to-have (входят в контракты волн)

1. Роль-middleware на V2-админ-роуты: mutation-группы (`activate policy`, `mark-paid`, конфиг) —
   `calculator.role:owner`; read-группы — `calculator.role:owner,finance`. Роль в route middleware,
   не в комментариях; negative-тесты обязательны.
2. IDOR-скоуп cabinet-эндпоинтов оплаты со счетов: заказ резолвится через аутентифицированного
   члена; negative-тест на чужой order id.
3. Точность PV: `decimal(18,6)` во всех V2-таблицах; `matched_bv_cents` из T03 — единственный
   денежный вход T06 (T06 не пере-выводит BV из PV).
4. `PaidOrderV2Pipeline` (T03) — единственная точка расширения пост-оплаты; T07 регистрирует
   `ReferralBonusStep`, `markPaid` напрямую никто не правит.
5. Дисциплина локов: `pg_advisory_xact_lock(ACTIVATION_LOCK_KEY)` берёт только внешний
   оркестратор события (оплата/закрытие периода/возврат/cutover); внутренние сервисы —
   `assertLockHeld()`.
6. Внутренне-финансируемые заказы (ОС ≤70% + БС): разрешены, BV полноценный (по спеке);
   конфиг-флаг `internal_funding_full_bv=true` — рычаг на будущее.
7. UNALLOCATED-строки глобального пула: partial unique index на `(pool_id)` where member_id is null
   (либо sentinel), дедуп обязателен.

- **Worked example в T11:** числовой сквозной пример калибровки (base 10000 / Σ 10000 →
  factor_bps = 6000) — в тестах и в отчёте админа.
- **Подписи владельца (зафиксированы):** сгорание сматченного PV сверх капа — решено владельцем
  на Гейте A («Сгорает»); лаг доступности структурной премии до закрытия месяца (вариант A,
  ~2–6 недель от события) — принят как дефолт, владелец уведомлён явно и не возразил
  (лог сессии 2026-07-12); откат на буквальное «1/16» возможен конфигом расписания.

## Контракт-чеки волн W2+ (из fixloop W1, обязательны)

1. НС кредитуется ТОЛЬКО через `LedgerV2::credit()` с явным `accrualMonth` (T06 при закрытии H2,
   тикающем 1-го числа, передаёт месяц явно); сырой `post()` для НС запрещён; residue-guard.
2. При появлении дебетов НС (T12 возвраты) — декрементировать месячные бакеты `ns_month`.
3. Канонический порядок локов: Order → reservation/account; ACTIVATION_LOCK — только оркестратор.

## MF-W4-1 (ревью W4) — единая истина «фактически выплаченной структурной»

**Дата:** 2026-07-12 · **Ревью:** docs/reviews/2026-07-12-mh-full-plan-wave4-review.md (CRITICAL, деньги)

Канон стыка T11→T08 (закрывает разрыв «лидерский считался от некалиброванного after_cap»):

- **Одно каноническое значение выплаченной структурной премии** на строку
  `v2_structure_bonuses` = `net_cents` = `intdiv(after_cap_cents × factor_bps, 10000)`
  (per-member floor: калиброванная сумма участника раздаётся по его строкам методом
  наибольшего остатка, Σ строк = `intdiv(Σ after_cap × factor, 10000)` — без per-row дрейфа).
- **Владелец записи — T11** (`PoolCalibrationService::calibrateMonth`): пишет `net_cents`
  (+ `pool_coefficient` = factor/10000, `pool_adjustment_cents` = after_cap − paid) на
  month-close шагом order 500, СТРОГО ДО шага лидерского order 800 (DEC-053). До T11
  `net_cents = after_cap` (T06 default) — это временное значение, не «выплачено».
- **Читатели единой истины:** база лидерского T08 (`StructureBonusBaseSource` → `net_cents`)
  берёт калиброванное значение как есть и factor повторно НЕ применяет.
- **НС→ОС (T04) двойного factor НЕ создаёт:** он читает СЫРОЙ НС из ledger (закредитован
  `after_cap` на half-month posting — order 900, до месячной калибровки) и множит factor
  сам (`intdiv(raw × factor, 10000)`, per-member floor). `net_cents` он не читает, поэтому
  калибровка колонки после posting'а на его перевод не влияет. Оба консумента дают одно и
  то же выплаченное `after_cap × factor` из своей raw-базы — расхождения нет.
- **Fail-safe порядка:** posting структурной — half-month-only (`supports()==HALF_MONTH`);
  калибровка и лидерский — month-only. Они не сосуществуют в одном close-run, поэтому NS
  всегда закредитован сырым after_cap до того, как T11 перепишет `net_cents`.
- **Инвариант «никто не читает after_cap как выплаченное»** держится: единственный
  after_cap-как-база остаётся только внутри `monthlyUsedCents` (лимит месячного капа —
  это by design до-пульная величина, DEC-053), не в payout-путях.
- Регресс: `LeadershipCalibrationIntegrationTest` — боевой month-close (T11→T08) на общих
  строках, assert `base = after_cap × factor` (красный до фикса: переплата на 1−factor/10000).
