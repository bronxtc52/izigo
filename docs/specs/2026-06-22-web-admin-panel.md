# ТЗ: Веб-админ-панель IziGo (admin.izigo.adarasoft.com)

**Дата:** 2026-06-22 · **Статус:** Гейт 1 (контекст/ТЗ) · **Конвейер:** /armanda

## Цель
Широкая веб-админ-панель для управления MLM-платформой IziGo на отдельном домене
`admin.izigo.adarasoft.com`. Две движущие причины:
1. **Админку нельзя показывать в Telegram Mini App** — её надо полностью убрать из Mini App
   (вкладка «Админ» в `MiniAppShell`/`MiniAppAdmin`) и дать только в вебе. → **пункт следующего релиза.**
2. Нужен полноценный пульт управления: пользователи, финансы, операции, выплаты, продукты и —
   главное — **настройки маркетинг-плана** (полное редактирование боевого комп-движка).

## Пользователи и роли
RBAC уже есть (`Member::roles()`, `RoleMiddleware`): **owner / finance / leader / support**.
Веб-админка переиспользует те же роли и гейты эндпоинтов как есть.
- **owner** — всё, включая роли и маркетинг-план.
- **finance** — финансы, выплаты, KYC.
- **support** — пользователи (чтение), продукты, заказы.
- **leader** — пользователи своего поддерева (read-only, `leader_scope_member_id`).

User stories (примеры):
- *Как owner* я вхожу через Telegram-аккаунт в браузере и попадаю в админку с полным доступом.
- *Как owner* я меняю проценты бинарного/реферального/лидерского бонуса, пороги рангов и PV
  пакетов — и новые активации считаются по новым значениям (прошлое не пересчитывается).
- *Как finance* я вижу очередь заявок на вывод, баланс партнёра и одобряю/отклоняю/выплачиваю.
- *Как support* я завожу/архивирую товары и меняю статус заказов.

## Ключевые архитектурные решения (Гейт 1)

### 1. Аутентификация — Telegram Login Widget (НЕ initData)
Вся платформа сейчас авторизуется только Telegram initData, которого в браузере нет.
Решение: **Telegram Login Widget** на странице входа веб-админки.
- Поток: браузер → кнопка «Войти через Telegram» → Telegram отдаёт **подписанные данные**
  (`hash` по HMAC-SHA256 с `bot_token`) → backend валидирует подпись (переиспользуем логику
  из `TelegramInitData`/`MiniAppAuth`) → резолвит `Member` по `telegram_id` → выдаёт
  **Sanctum personal access token** (или сессию) для последующих запросов.
- Новый backend-слой только на **аутентификацию** (резолв member из web-запроса). **RBAC
  (`RoleMiddleware`) и сами admin-эндпоинты — без изменений.**
- Доступ в админку — только members с непустым набором ролей (как `isAdmin` сейчас).
- Никаких паролей в БД (согласуется с дисциплиной секретов).
- Требование: HTTPS-домен прописать боту через `setDomain` (`admin.izigo.adarasoft.com` готов, cert green).

### 2. Размещение — тот же Next.js-апп, роутинг по хосту
Веб-админка живёт в существующем `mh-calc-frontend-main` (тот же ACA `ca-izigo-frontend`,
куда уже указывает `admin.izigo.adarasoft.com`).
- Middleware Next.js разводит по `Host`: `admin.izigo.adarasoft.com` → веб-роуты `/admin/*`;
  `izigo.adarasoft.com` → Mini App. Прямой заход на админ-роуты с не-admin хоста — 404/redirect.
- Переиспользуем существующие компоненты `src/views/admin/*` (MembersList, MemberCard,
  PlanSettings, AdminWithdrawals), заменив auth-слой: `initDataApi` (`X-Telegram-Init-Data`)
  → новый `webAdminApi` (`Authorization: Bearer <sanctum>`). Тот же набор эндпоинтов.
- Один деплой, один CI, DNS/cert уже готовы.

### 3. Маркетинг-план — полное редактирование боевого ядра
Сейчас почти все параметры захардкожены в `Modules/Calculator/Domain/Plan/IziGoPlanFactory.php`
и продублированы в легаси-симуляторе. Решение:
- **Единый источник правды = доменное ядро.** Выносим параметры в `plan_settings` (key/value JSON),
  `EloquentPlanRepository::load()` читает их поверх дефолтов фабрики (точка расширения уже заложена).
- **Легаси-симулятор `CalculatorService/getConfig()` — вне скоупа** (это калькулятор-симулятор,
  не деньги). Риск рассинхрона задокументировать; не трогаем в этой итерации.
- **Безопасность изменений:** каждое изменение плана пишется в **аудит-лог** (кто/что/когда,
  старое→новое значение); изменения применяются **вперёд** — прошлые начисления не пересчитываются
  (новые активации считаются по актуальному снимку плана). Валидация диапазонов (проценты 0–100, ≥0).

## Границы итерации (что НЕ входит)
- **Не трогаем легаси-симулятор** `CalculatorService/getConfig()` и таблицы `calculator_*` volumes —
  только боевое ядро. Унификация двух движков — отдельная задача.
- **Не реализуем новые типы бонусов** (матрица/unilevel/fast-start/пулы) — только настройка
  существующих 4 типов.
- **Не пересчитываем прошлые начисления** при смене плана (forward-only).
- **Mini App функционал** (кабинет/магазин партнёра) не меняем, кроме **удаления вкладки «Админ»**.
- Email/password-логина не делаем (только Telegram Login Widget).

## Открытые риски
1. **Редактирование живого комп-движка** влияет на реальные деньги → обязательны валидация,
   аудит, forward-only и аккуратное ревью (Гейт 4).
2. **Рассинхрон боевого ядра и легаси-симулятора** после выноса в БД — симулятор останется на
   хардкоде. Зафиксировать как известное ограничение.
3. **Telegram Login Widget** требует корректного `setDomain` у бота и HTTPS — проверить на этапе кодинга.
4. Объём большой → на Гейте 2 (план) разбить на MVP→v2, чтобы не утонуть.
