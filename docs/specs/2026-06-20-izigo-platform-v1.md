# ТЗ: IziGo — MLM-платформа v1 (MVP)

**Дата:** 2026-06-20
**Основано на:** опросах (4 раунда), `roadmap.md`, `mlm_platform.md`, разведке калькулятора.
**Статус:** черновик ТЗ на согласование.

## 1. Цель

Превратить текущий калькулятор IziGo (симулятор бинарного маркетинг-плана) в
**рабочую MLM-платформу-биллинг**: реальные партнёры, покупка пакетов, корректное
реал-тайм начисление бонусов в кошелёк и вывод в USDT. Канал — web + Telegram (бот + Mini App).
Главный приоритет качества — **корректность и безошибочность расчётов (это биллинг)**.

## 2. Зафиксированные решения (из опросов)

| Параметр | Решение |
|---|---|
| Модель бизнеса | **Товарный MLM** (объём PV/BV с покупок) |
| Масштаб v1 | **MVP**, быстрый запуск, наращивать потом |
| Валюта | **только USD** (выплаты/учёт; USDT ≈ USD 1:1) |
| Каналы | **Web + Telegram-бот + Telegram Mini App** (партнёр и клиент работают в TG) |
| Стек | **Laravel + Next** (+ Telegram Mini App на Next) |
| География/языки | СНГ/Центр. Азия, 6 локалей (az/kk/ky/mn/ru/uz) |
| Комп-план | **Только бинарный** |
| Бонусы | binary + referral + leader + rank **+ глобальный (пул)** |
| Расчёт бонусов | **Реал-тайм** (мгновенно по событию) |
| Начисление/выплата | Авто-начисление в кошелёк; **вывод по заявке с одобрением админа** |
| Объём (volume) | **Только PV** (BV убран). **1$ = 0.9 PV** (10% — некомиссионная маржа). Бонусы = % от PV, выплата USD по **1 PV = $1** |
| Вывод средств | **USDT** |
| Входящая оплата | **USDT + карта** |
| Размещение в дереве | **Настраиваемо** (ручное спонсором / авто-спилловер — переключатель компании) |
| KYC | **Лёгкий** (email/телефон/Telegram-аккаунт, без документов) |
| Покупка | **Фикс-пакеты** Bronze $100=90PV / Silver $200=180PV / Gold $600=540PV, без полного каталога |
| Роли админки | **Полный RBAC** (владелец / финансы / лидер / саппорт) |
| Стартовые данные | **С нуля** (без миграции) |

## 3. Акторы и роли

- **Клиент/партнёр** — регистрируется (web/Telegram), покупает пакет → активируется,
  строит сеть, видит доход и команду, выводит USDT. (В товарном MLM «клиент» = партнёр
  с пакетом; чистых покупателей-неучастников в v1 не выделяем.)
- **Админ-RBAC:** Владелец (всё), Финансы (выплаты/отчёты), Лидер (своя ветка/регион),
  Саппорт (просмотр/помощь). Права — конфигурацией, без кода.

## 4. Функциональные модули v1

### 4.1 Аккаунты, авторизация, лёгкий KYC
- Регистрация/вход: email+пароль (есть) **и вход через Telegram** (Telegram Login / initData
  валидация для Mini App). Привязка Telegram-аккаунта к партнёру.
- Лёгкий KYC: подтверждение email/телефона; верификация Telegram. Без документов в v1.
- Реф-ссылка (и deep-link в Telegram-бот) для приглашения.

### 4.2 Сеть и генеалогия (binary)
- Реальные сущности **member**; два графа: placement (бинар) и sponsorship (ЛП) — как в ядре.
- **Размещение настраиваемо**: режим компании — ручное (спонсор выбирает позицию) ИЛИ
  авто-спилловер (в слабую/левую ногу). Переключатель в настройках.
- Нормализованное хранение дерева (Postgres `ltree`/closure table) — НЕ JSON-блоб.
  Конкурентная безопасность (локи/версии) при размещении и пересчёте.
- Визуальное дерево команды (переиспользуем react-d3-tree из калькулятора).

### 4.3 Покупка и активация пакетов
- Пакеты: **Bronze $100 = 90 PV / Silver $200 = 180 PV / Gold $600 = 540 PV** (PV = 0.9×цена).
- Оформление покупки → оплата **USDT или картой** → подтверждение оплаты → активация пакета →
  событие «активация» в биллинг.
- USDT: приём через крипто-шлюз (сеть уточнить, напр. TRC20); карта: эквайринг (провайдер уточнить).
- Ручное подтверждение оплаты админом — как fallback (offline).

### 4.4 Биллинг-ядро (КЛЮЧЕВОЙ модуль — максимум строгости)
- **Событийная архитектура:** денежное/объёмное событие (активация пакета, покупка) →
  идемпотентные обработчики начислений (idempotency-key = id события, **exactly-once**).
- **Ledger с двойной записью**, неизменяемые проводки, **полный аудит-трейл** каждого начисления
  (кто/что/почему/формула). Деньги — **decimal** (не float!).
- **Реал-тайм пересчёт** при событии: объёмы PV (личный/групповой) вверх по дереву → ранги → бонусы.
  Все бонусы считаются на **PV** (BV убран); деньги — USD по 1 PV = $1:
  - **Binary** — пайринг по малой ветке PV, carryover/flush (логика калькулятора, база BV→PV).
  - **Referral** — % от PV пакета по линии спонсоров (глубина 2).
  - **Leader** — bonus-on-bonus с rank-compression.
  - **Rank** — разовый при достижении ранга.
  - **Global (пул)** — % от общего оборота для топ-рангов. ⚠️ Пул по своей природе
    **периодический** (нужен оборот за период) — единственное исключение из реал-тайма:
    считается на закрытии периода. **Правила/период/ранги — уточнить (см. §9).**
- **Reconciliation:** сверка ledger ↔ кошельки ↔ выплаты; отчёт расхождений.
- **Тесты-инварианты** на формулы и ledger (golden-эталон из калькулятора + новые) —
  обязательны до прода.

### 4.5 Кабинет партнёра (web + Telegram Mini App, единый бэкенд)
- Дашборд дохода: разбивка по 5 типам бонусов + **логика расчёта** каждой выплаты, история по периодам.
- Прогресс рангов: текущий ранг, условия следующего, визуальный прогресс-бар.
- Дерево команды (граф), поиск/фильтр веток.
- История покупок/активаций.
- Кошелёк: баланс USD, **заявка на вывод USDT**, статус заявок.
- Профиль, реф-ссылка, KYC-статус.

### 4.6 Админ-портал + RBAC
- Управление участниками (поиск/сегментация по рангу/региону/активности).
- **RBAC** (владелец/финансы/лидер/саппорт), права в настройках.
- Настройка бонусов/порогов рангов/процентов/режима размещения.
- **Очередь выводов**: одобрение/отклонение заявок USDT, лог, сверка.
- Подтверждение входящих оплат (fallback), управление пакетами.
- Отчёты: продажи/комиссии по рангам/активность/удержание; экспорт Excel/PDF; период-клозинг (для глобального бонуса) с аудитом.

### 4.7 Кошелёк и выплаты
- Внутренний кошелёк (USD-баланс) — авто-кредит начислений.
- Вывод: партнёр создаёт заявку (USDT-адрес) → админ (роль «финансы») одобряет → выплата.
  Авто-исполнение USDT — позже; в v1 одобрение обязательно.

### 4.8 Telegram (бот + Mini App)
- **Бот:** регистрация/онбординг, уведомления (новый реферал, начислен бонус, статус вывода,
  достижение ранга), быстрые команды (баланс, реф-ссылка), deep-link приглашения.
- **Mini App:** полноценный UI кабинета внутри Telegram (initData-валидация по HMAC).

## 5. Нефункциональные требования

- **Корректность биллинга — приоритет №1:** exactly-once начисления, идемпотентность,
  ledger-инварианты, аудит, reconciliation, покрытие тестами. Никаких float в деньгах.
- **Безопасность:** RBAC, 2FA для админов, шифрование at-rest/in-transit, аудит-лог,
  валидация Telegram initData, защита вывода (anti-fraud, лимиты, подтверждение).
- **Observability:** Sentry (self-host), Azure-теги project/env/owner, secrets — Key Vault,
  server-watchdog. CI/CD + контейнеризация (ACA).
- **Локализация:** 6 языков (есть), единая валюта USD.
- **Производительность:** дерево/пересчёт без таймаутов на целевом объёме MVP; Postgres.

## 6. Архитектура / стек

- Backend: **Laravel 10** (модульный), **PostgreSQL** (миграция с MySQL; `ltree`/closure для дерева).
- Frontend: **Next 14** (web-кабинет + Telegram Mini App), antd, react-d3-tree.
- Telegram-бот: отдельный воркер (aiogram/grammY или Laravel-бот) на ACA.
- Биллинг: событийный, очереди (Laravel queues) для обработчиков начислений, ledger в БД.
- Деплой: Azure Container Apps; секреты — Key Vault.

## 7. Переиспользуем из калькулятора

- Доменную **комп-логику** (binary/referral/leader/rank, ранги, PV/GV) — извлечь в чистый
  тестируемый модуль (golden-тесты как эталон корректности).
- Пакеты Bronze/Silver/Gold, ранги, 6 локалей, react-d3-tree, локальную авторизацию.
- Сам калькулятор — оставить как маркетинг/онбординг-витрину.

## 8. Вне scope v1

- Полный каталог товаров/корзина (только фикс-пакеты), autoship.
- Полноценный док-KYC/Sumsub, авто-исполнение USDT без одобрения.
- Мобильные нативные приложения (есть Telegram Mini App + web).
- AI-аналитика, геймификация, replicated sites, CRM/ERP-интеграции — позже по roadmap.
- Мультивалютность/мультистрановой комплаенс (только USD на старте).

## 9. Решения по ранее открытым вопросам (рекомендованные дефолты — confirm бизнесом)

1. **Глобальный бонус:** пул = **3% от общего BV-оборота компании**, накопление помесячно,
   **выплата раз в квартал**, распределение между топ-рангом долями (выше ранг → больше долей).
   Соответствует тексту UI-модала «Глобальный бонус». Единственный периодический бонус
   (real-time невозможен — нужен оборот за период). **Требует верхнего ранг-тира (Director+)**
   — добавить в справочник рангов (сейчас максимум `manager_silver`).
2. **Бинар:** оставляем логику калькулятора (флэт 5% малой ветки, carryover/flush), но **база BV→PV**
   (BV убран). Заложить **настраиваемый cap** (потолок бинар-выплаты/период), дефолт выкл;
   рекомендуется включить против неконтролируемых обязательств.
3. **Платёжные провайдеры:** USDT — **сеть TRC20**, шлюз **Cryptomus** (приём+выплаты; альт. NOWPayments).
   Карта — за **адаптером PaymentProvider**; v1 primary = USDT, карта fast-follow через PSP под юрлицо
   компании (зависит от банкинга — нужен ввод бизнеса).
4. **Вывод:** мин. **$30**; комиссия = сетевая + ~$1–2/2%; 1 активная заявка; настраиваемые
   дневной/недельный лимиты; суммы > $1000 — доп.review. Anti-fraud: одобрение админом + **2FA**,
   подтверждение/whitelist USDT-адреса, velocity-чеки, **hold 24–72ч для новых аккаунтов**,
   флаг переиспользования адреса.
5. **Цены/объём:** **BV убран**, только PV. **1$ = 0.9 PV** (PV = 0.9×цена; 10% маржа).
   Круглые цены: Bronze **$100 = 90 PV** / Silver **$200 = 180 PV** / Gold **$600 = 540 PV**.
   Бонусы = % от PV; выплата в USD по **1 PV = $1**. ⚠️ Проверить, что сумма бонус-% < комиссионного
   объёма (90 PV на $100) — устойчивость плана (связано с cap в п.2).
6. **Размещение:** дефолт — **ручное спонсором** (свободный слот своего поддерева, с валидацией);
   авто-спилловер (опц.) — в **слабую ногу**, breadth-first.
7. **Клиент vs партнёр:** **единый актор — партнёр**. Покупка пакета = активация = позиция в дереве.
   Розничные покупатели-неучастники — не в v1.

### Остаются на бизнес-подтверждение (не блокируют старт Фазы 0–1)
- Карточный PSP (зависит от юрлица/банкинга).
- Экономика плана: устойчивость (сумма бонус-% vs комиссионный объём 90 PV/$100) и значение cap.
- Точные доли/условия глобального бонуса и название/пороги ранга Director+.

## 10. Критерии приёмки v1

- Партнёр регистрируется (web/Telegram), покупает пакет (USDT/карта), активируется.
- Реал-тайм: покупка/активация → корректные начисления всех бонусов в кошелёк, видны в кабинете
  с логикой расчёта; ledger сходится (reconciliation без расхождений).
- Дерево команды отображается; ранги пересчитываются.
- Партнёр создаёт заявку на вывод USDT → админ (финансы) одобряет → статус обновляется.
- Админ-RBAC: роли видят только разрешённое; настройка бонусов/режима размещения работает.
- Telegram: бот шлёт уведомления, Mini App показывает кабинет.
- Тесты биллинга (инварианты/идемпотентность/golden) — зелёные; Sentry/CI/секреты в KV настроены.
