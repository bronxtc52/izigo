# ТЗ: Telegram-only авторизация и регистрация (IziGo)

**Дата:** 2026-06-21
**Тип:** фича/рефакторинг auth-слоя (backend + frontend + БД). Прогон через `/armanda`.
**Статус:** Гейт 1 — ТЗ (на подтверждении).

## 1. Цель

Единственный способ регистрации и входа в платформу — **Telegram**. Других механизмов
(email+пароль, SSO, телефон) быть не должно. При первом запуске человек регистрируется
через Telegram, и его аккаунт навсегда связан с этим `telegram_id`. Вся последующая
авторизация (в т.ч. «веб») тоже идёт через Telegram.

Сейчас в проекте **две независимые идентичности**, не связанные между собой:
- web: `calculator_users.email` + bcrypt-пароль → токен `CalculatorUserToken`;
- Telegram Mini App: `members.telegram_id` по initData (HMAC), без линковки к web-аккаунту.

Один человек, зашедший двумя путями, получает **два разных узла в дереве**. Это устраняем:
единственная идентичность — `telegram_id`.

## 2. Решения Гейта 1 (подтверждены пользователем)

1. **Веб-вход:** отдельного входа в браузере НЕТ. Кабинет и админка доступны ТОЛЬКО как
   Telegram Mini App (работает и в Telegram Desktop — десктоп-сценарий закрыт). Прямой URL
   в браузере → экран «Откройте через Telegram». Telegram Login Widget НЕ используем.
2. **Данные:** beta-данные сбрасываем. Миграции на email/password откатываем/удаляем,
   старые email-аккаунты не мигрируем. Чистый старт: единственная идентичность = `telegram_id`.
3. **Бутстрап админа:** владельцы заданы списком `telegram_id` в конфиге/Key Vault
   (`OWNER_TELEGRAM_IDS`). При первом входе такой участник автоматически получает роль `owner`
   и дальше раздаёт роли остальным из админки. Список не хранится в коде.
   - **Первый владелец:** `telegram_id = 201374791` (@bronxtc52), без email.

## 3. Пользователь и сценарии (user stories)

- **US-1 (новый партнёр):** Открываю бота `@Izigopro_mlm_bot` → «Открыть IziGo» → Mini App.
  Я автоматически зарегистрирован по `telegram_id` (initData), помещён в дерево по спонсорской
  реф-ссылке (`start_param`). Никаких форм email/пароля не вижу.
- **US-2 (повторный вход):** Открываю Mini App снова → меня узнают по `telegram_id`, сразу вижу
  свой кабинет. Логина/пароля нет.
- **US-3 (браузер вне Telegram):** Захожу на сайт в обычном браузере → вижу «Откройте через
  Telegram» со ссылкой на бота. Войти иначе нельзя.
- **US-4 (владелец):** Мой `telegram_id` в `OWNER_TELEGRAM_IDS`. При первом входе в Mini App
  получаю роль `owner`, вижу админ-раздел, могу назначать роли (finance/leader/support) другим
  участникам по их `telegram_id`/поиску.
- **US-5 (админ-действия):** finance/leader/support видят админ-разделы по своим ролям (RBAC),
  тоже только через Telegram Mini App.

## 4. Ограничения

- Стек как есть: Laravel (backend) + Next App Router (frontend) + grammY-бот; Postgres + ltree.
- Идентичность и доступ — по initData (HMAC-SHA256, secret=HMAC("WebAppData", bot_token)),
  механизм уже реализован (`TelegramInitData`, `MiniAppAuth`) — переиспользуем, не изобретаем.
- Токен бота и `OWNER_TELEGRAM_IDS` — ТОЛЬКО из Key Vault/конфига, не в git.
- Тесты обязательны (auth-флоу, бутстрап owner, отсутствие альтернативных входов).

## 5. Что входит в итерацию

- Удаление email/password регистрации и входа: контроллер `LocalAuthController`, маршруты
  `/auth/register` и `/auth/login`, `LocalAuthService` (email-ветка), миграция пароля, форма
  `LocalAuth.js` и web-гейт по `CalculatorAuthToken`.
- Единая идентичность по `telegram_id` для кабинета И админки (Mini App как единственная точка).
- RBAC поверх Telegram-идентичности (роли привязаны к участнику по `telegram_id`, не к email).
- Бутстрап `owner` из `OWNER_TELEGRAM_IDS` при первом входе.
- Frontend: убрать `(cabinet)`/`(admin)` web-вход; админ-UI доступен внутри Mini App по роли;
  браузерный заход → «Откройте через Telegram».
- Сброс/чистка beta-данных и связанных миграций.

## 6. Что НЕ входит (границы)

- Telegram Login Widget для браузера (решено: не делаем).
- Миграция/линковка старых email-аккаунтов (данные сбрасываются).
- Смена реф-механики/спонсорства (остаётся `start_param`-дефолт).
- Новый план «NEXUS» (`docs/Binar_Company_2027.docx`) — отдельная инициатива, не здесь.
- Веб-выплаты/Web3 — вне этой итерации.

## 6a. Легаси-витрина калькулятора (осознанное решение)

Публичный калькулятор-витрина (анонимный расчёт дерева по `/packages`,`/ranks`) продолжает
работать БЕЗ входа. Персональная фича «сохранить структуру в свой аккаунт»
(`/calculator/structure/*` под токеном) выключается: токен раньше выдавал только email-вход,
а он удалён. Возвращать issuer токена НЕ будем — это был бы второй способ входа, что прямо
противоречит «only Telegram». До перехода вся витрина и так была за email-логином (GlobalContext
форсил форму) — т.е. для конечного пользователя анонимный калькулятор стал ДОСТУПНЕЕ, а не наоборот.
Персональное сохранение структур — вне области (продукт = Telegram MLM-платформа).

## 7. Архитектурные решения (подтверждены)

- **Схлопываем `CalculatorUser` полностью.** Единственная сущность идентичности — `Member`
  (ключ `telegram_id`). Роли/язык/валюта переезжают на `Member`. Таблицы `calculator_users`,
  `calculator_user_tokens` и пивот `role_user` удаляются; RBAC — через `member_roles`.
- **Доступ — по initData на каждый запрос** (как в Mini App), bearer-токен
  `CalculatorUserToken` упраздняется.
