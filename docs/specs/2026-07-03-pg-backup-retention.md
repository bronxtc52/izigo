# Бэкапы Postgres IziGo: retention + logical-дампы + runbook восстановления

**Дата:** 2026-07-03 · **Конвейер:** /armanda · **Источник:** отложенный пункт аудита
готовности к коммерческому запуску (`docs/reviews/2026-07-02-commercial-launch-readiness-audit.md`).

## Цель

Прод-БД (`izigo-pg-beta`, PG16, единственная база `izigo`) держит деньги (ledger, платежи,
выводы). Сейчас защита — только дефолтный PITR 7 дней в том же регионе, без geo-избыточности,
без процедуры восстановления. Нужно: удлинить окно PITR, завести независимый кросс-регионный
logical-бэкап и записать проверенный runbook восстановления.

## Факты (разведка 2026-07-03)

- `izigo-pg-beta` @ rg-izigo-beta-neu (northeurope): B1ms, 32GB, autoGrow off, HA off,
  retention **7д**, geo-redundant **Disabled** — включить на живом сервере НЕЛЬЗЯ (only at create).
- Firewall: только `AllowAllAzureServices` (0.0.0.0). Доступ: user `izigo`, пароль
  `izigo--beta--DB-PASSWORD` в KV, sslmode=require.
- mh-central (westeurope): `pg_dump` 16, MSI читает KV, ночной cron-паттерн есть (KV-backup 03:00),
  финальные дампы проектов исторически живут в `~/backups/`.

## Состав работ

1. **Retention PITR 7 → 14 дней** — `az postgres flexible-server update --backup-retention 14`.
   Онлайн, без рестарта. Стоимость: бэкап-сторадж бесплатен до 100% provisioned (32GB);
   БД беты крошечная — остаёмся в бесплатном объёме. 🔴 Красная зона (изменение прод-ресурса) —
   выполняется по утверждению этого ТЗ.
2. **Ночной logical-дамп на mh-central** — скрипт `ops/backup-izigo-pg.sh` в репо
   (деплой копированием на VM): `az login` уже MSI → пароль из KV → `pg_dump -Fc` базы `izigo`
   → `~/backups/izigo-pg/izigo-YYYYMMDD-HHMMSS.dump` (chmod 600, каталог 700) → ротация: хранить
   **30 дней** → верификация свежего файла `pg_restore --list`. Cron **03:30** ежедневно,
   лог в `~/backups/izigo-pg/backup.log`. Контингенция: если Azure-services-правило не пускает VM —
   добавить firewall-правило `AllowMhCentral` (51.138.74.197), покрывается этим же утверждением.
3. **Runbook** `ops/izigo-pg-backup-restore.md`: (a) PITR-восстановление во временный сервер +
   перенос/repoint, (b) восстановление из logical-дампа, (c) верификация после восстановления
   (health, счётчики members/payments/ledger_entries), (d) RPO/RTO-таблица по сценариям (потеря данных / регион /
   человеческая ошибка), (e) ограничение geo-redundant + чем компенсируем, (f) чек-лист
  restore-drill (повторять квартально).
4. **Боевой restore-drill сейчас** — свежий дамп восстановить в локальный docker PG16
   (отдельная БД на izigo-test-pg) и сверить с продом количество таблиц и строк в
   `calculator_members`/`ledger`-таблицах. Runbook считается проверенным, а не теоретическим.
5. Доки: пометка о бэкапах в `binar-mlm/CLAUDE.md` (раздел про прод) — одна-две строки.

## Границы (что НЕ делаем)

- Движок/код приложения не трогаем; никакого кода в Laravel.
- HA и geo-redundant backup НЕ включаем (пересоздание сервера / ×2 стоимости — отдельное решение).
- Новые Azure-ресурсы НЕ создаём; storage для дампов — диск mh-central (как у других проектов).
- Дампы содержат PII → живут только на mh-central (600/700), наружу не уходят, в git не попадают.

## Критерии приёмки (да/нет)

1. `az ... show` возвращает `retentionDays: 14`.
2. `ops/backup-izigo-pg.sh` отработал на mh-central: файл дампа существует, `pg_restore --list`
   читает его без ошибок, права 600.
3. Дамп восстановлен в локальный PG16: восстановление завершилось без ошибок, число таблиц
   и `count(*)` ключевых таблиц (`members`/`payments`) совпадают с продом на момент дампа.
4. `crontab -l` на mh-central содержит строку 03:30 для izigo-pg; ротация подтверждена
   (файл старше 30 дней удаляется — проверка тач-файлом).
5. Runbook существует и покрывает пункты (a)–(f); PITR-процедура выписана командами
   (сам PITR-прогон НЕ выполняем — создание сервера стоит денег; это зафиксированная граница).
6. Правка CLAUDE.md есть; изменения уезжают feature-веткой + PR (docs/ops-only, `[skip ci]`
   не нужен — paths-ignore не покрывает ops/*.sh, но деплой-джобы гейтятся needs:test — прогон
   CI безвреден).

## Допущения (решены сам, зафиксированы)

- Retention 14, а не 35: длиннее окно — больше смысла при реальном трафике; на бете 14
  достаточно, поднять до 35 — одна команда, когда пойдут живые деньги.
- Ротация локальных дампов 30 дней без weekly/monthly-ярусов — объём копеечный, усложнение
  не окупается.
- Дампы не шифруем поверх (диск VM — наша граница доверия, там уже живут финальные дампы
  других проектов и бэкапы KV).
