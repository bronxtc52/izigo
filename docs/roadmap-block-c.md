# Блок C (адаптация из MHG) — план работы под 3 отдельных /armanda-цикла

> Источник идей: `docs/handoff-mhg-comparison.md`. Завтра запускать **каждый цикл отдельным
> `/armanda`** (один фокус на сессию). Порядок ниже — с учётом зависимостей.
>
> **Ограничения (неизменны во всех циклах):** движок бонусов `Modules/Calculator` НЕ трогаем
> (только читаем выходы); валюта только USD; авторизация и платёж только Telegram. См. memory
> `izigo-core-constraints`.

## Что уже есть в коде (фундамент, найдено разведкой)
- `Modules/Calculator/Services/Telegram/TelegramNotifier.php` + `TelegramNotifications.php` —
  бэкенд УЖЕ умеет слать сообщения в Telegram (фундамент для уведомлений/рассылок).
- Бот `mh-calc-bot` — Node long-polling (`src/bot.js`, `messages.js`).
- Таблиц/моделей notifications / outbox / tickets / feature-flags — НЕТ (всё greenfield).
- У `Member` НЕТ полей co-partner / heir / phone — это новые аддитивные данные.
- Веб-админка: `src/views/admin/web/` + `webApi.js` (Bearer Sanctum). Mini App: `src/views/miniapp/`
  + `api.js` (initData). Роли: owner / finance / support / leader.

## Рекомендованный порядок (зависимости)
Уведомления+outbox — это backbone, на который опираются и тикеты (пуш «новый ответ»), и рассылки.
Поэтому **Цикл 1 = Коммуникации/outbox**, затем тикеты, затем ops.

---

## 🔁 Цикл 1 (/armanda #1) — Коммуникации: центр уведомлений + рассылки + outbox
**Фокус:** надёжная доставка уведомлений участнику и админ-рассылки через Telegram-бот.

**Состав:**
- **Outbox-паттерн** — таблица `notification_outbox`; внешние эффекты (Telegram-пуши) пишутся в
  outbox в транзакции, обрабатываются воркером/планировщиком (Laravel scheduler — см. memory
  `izigo-aca-laravel-scheduler`: нужен `schedule:work` в start.sh), идемпотентно, с ретраями.
- **Доставка** — через существующий `TelegramNotifier` (отправка по `telegram_id` участника).
- **Центр уведомлений (Mini App)** — инбокс: список уведомлений, непрочитанные, отметка
  прочитанным. Новая вкладка/иконка-колокольчик.
- **Событийные уведомления** — хуки на доменные события (старт: статус выплаты; позже — новый
  ответ в тикете из Цикла 2). Шаблоны текста с нормализацией (см. правило md→Telegram-HTML).
- **Админ-рассылки** — экран: текст + сегмент (все / по статусу / по рангу) → постановка в outbox
  пачкой; rate-limit; превью охвата (dry-run «сколько получат»); аудит.

**Решить в Гейте 1 этого цикла:**
- Сегменты рассылок: все / по статусу (registered/active) / по рангу — что в MVP.
- Какие события шлют пуш в MVP (минимум: статус выплаты; задел под тикеты).
- Хранить ли уведомление в БД всегда (инбокс) + дублировать в Telegram, или Telegram-only.

**Не трогаем:** доставку SMS/email; движок; деньги.

---

## 🔁 Цикл 2 (/armanda #2) — Поддержка: тикет-система helpdesk (чат)
**Фокус:** участник пишет обращение из Mini App; в админке это чат-helpdesk оператор↔участник.
**Зависит от Цикла 1** (пуш «новый ответ» идёт через центр уведомлений/outbox).

**Состав:**
- **Модель:** `tickets` (member_id, subject, status, assigned_to, priority?, timestamps) +
  `ticket_messages` (ticket_id, author = member|operator, body, attachments?, read_at).
- **Статусы:** open / in_progress / resolved / closed; назначение оператору (роль `support`/owner).
- **Mini App:** создать тикет, список своих тикетов, чат-переписка по тикету.
- **Админка:** очередь тикетов (фильтр по статусу/назначению), чат-панель helpdesk, ответ,
  смена статуса, «взять на себя».
- **Уведомления:** новый ответ оператора → пуш участнику (через Цикл 1); новый тикет/ответ
  участника → уведомление операторам.

**Решить в Гейте 1 этого цикла:**
- Транспорт чата: **polling** (рекоменд., у нас нет websockets) vs WS/Reverb.
- Очередь: общая для всех операторов vs назначение на себя; SLA/приоритеты — нужны ли в MVP.
- Вложения (картинки) — в MVP или отложить (text-only).
- Кто оператор: роль `support` (+ owner) — подтвердить.

## Ops/инфра — разбит на отдельные мелкие циклы (чтобы не идти большим планом)
Пункт 6 разнородный, поэтому каждый под-пункт — **свой маленький `/armanda`** (один фокус).
Все независимы; порядок свободный, кроме монитора (после Цикла 1 — ему нужен outbox/очередь).

### 🔁 Цикл 3 (/armanda #3) — Feature-flags
- Рантайм-тоглы в БД (`feature_flags`: key / enabled / описание), читаются бэком и фронтом
  (эндпоинт + кэш); админ-экран вкл/выкл; гейтят показ новых фич (в т.ч. фич из Циклов 1–2, 4–7).
- **Малый, без зависимостей — хороший «разогрев».**
- Гейт 1: какие первые флаги завести; кэш-инвалидция; читает ли Mini App флаги публично.

### 🔁 Цикл 4 (/armanda #4) — i18n из админки
- Редактируемые переводы (таблица строк перевода), правка без релиза; фолбэк на статические
  locale-файлы.
- Гейт 1: какие экраны/языки в MVP; ключи переводов; кто правит (owner/support).

### 🔁 Цикл 5 (/armanda #5) — User-экспорт + PII-режим (карточка участника, админка)
- Экспорт данных участника (CSV/JSON) с аудитом выгрузки.
- PII (телефон/реквизиты/контакты) замаскированы в карточке; «Показать» только owner, каждый
  reveal — в аудит.
- Гейт 1: маска+reveal+аудит (рекоменд.) vs только экспорт vs отложить; формат экспорта; что
  считаем PII.

### 🔁 Цикл 6 (/armanda #6) — co-partner / наследник в профиле
- **Только информация** (контакты совладельца/наследника: ИО, телефон, доля). Без влияния на
  деньги/дерево/авторизацию (функционал = задел движка/выплат = исключено).
- Аддитивные поля/таблица, member-вводимые, показ в профиле (Mini App) и в карточке (админка).
- Гейт 1: подтвердить «только информация»; какие поля; одна запись или несколько наследников.

### 🔁 Цикл 7 (/armanda #7) — Мониторинг джоб/очередей/outbox (админка)
- Read-only: статус scheduler / queue / outbox — последние прогоны, ошибки, размер очереди,
  застрявшие записи. **После Цикла 1** (нужен outbox для мониторинга).
- Гейт 1: что показываем в MVP; ретрай застрявших из UI или только просмотр.

> Циклы 3–7 мелкие и независимые — при желании можно слить соседние (напр. 3+4 «инфра-тоглы»
> или 5+6 «данные участника»), но по умолчанию идём по одному, как просили.

## Завтра — как запускать (по одному `/armanda` на сессию)
1. **Цикл 1** — коммуникации + outbox (backbone). Деплой.
2. **Цикл 2** — тикеты-helpdesk (после Цикла 1). Деплой.
3. **Цикл 3** — feature-flags (малый «разогрев»).
4. **Цикл 4** — i18n из админки.
5. **Цикл 5** — user-экспорт + PII-режим.
6. **Цикл 6** — co-partner / наследник в профиле.
7. **Цикл 7** — монитор джоб/очередей/outbox (после Цикла 1).

Приоритет ценности: 1 → 2 → (3 flags) → 5 PII/экспорт → 7 монитор → 4 i18n → 6 профиль.
Каждый цикл: своё ТЗ в `docs/specs/<дата>-<тема>.md`, свой `plan.md`, рабочая ветка + PR,
тесты пропорционально риску, деплой в конце. Открытые вопросы по каждому циклу — выше, чтобы
Гейт 1 шёл быстро.
