# IziGo — роадмап развития из калькулятора в MLM-платформу

**Дата:** 2026-06-20
**Опирается на:** `mlm_platform.md` (ресёрч), разведку кода (калькулятор-симулятор).
**Принцип порядка:** зависимости → ценность → риск. Сначала фундамент и реальная модель
данных, потом то, чем партнёр пользуется ежедневно, потом деньги, потом рост.

## Где мы сейчас (стартовая точка)

- ✅ **Ядро комп-плана** (binary + бонусы referral/leader/rank, ранги, PV/GV, пакеты
  Bronze/Silver/Gold) — но в режиме **симуляции «что-если»**, дерево = JSON-блоб, деньги = float.
- ✅ Локальная авторизация (email+пароль), 6 локалей + валюты (отображение), бренд IziGo.
- ❌ Нет: git-репозитория, Sentry, CI/CD, контейнеризации, реальной сети/членов, заказов,
  ledger, кошелька, выплат, KYC, ролей/админки, мобайла.
- Стек: Laravel 10 + Next 14 + MySQL.

## ⚠️ Развилка, определяющая порядок (нужно решение)

**Модель бизнеса IziGo:**
- **(A) Товарный MLM** (PV с покупок товаров) → e-commerce и autoship становятся *ядром*,
  поднимаются в Фазу 3–4. (Пакеты «100/200/600 PV» намекают на это.)
- **(B) Инвест/реферальный** (объём от взносов/подписок, без товаров) → e-commerce почти не нужен,
  фокус на кошелёк/выплаты/инвест-лоты.

Роадмап ниже написан под **(A) товарный** как дефолт; для (B) — Фаза «Commerce» заменяется на
«инвест-лоты/подписки», остальное идентично. **Подтверди модель — уточню Фазы 4–5.**

---

## Фаза 0 — Фундамент и стандарты (1–2 нед) 🔧

Без этого нельзя в прод. Закрывает наши же глобальные правила (observability, секреты, CI).
- [ ] **git-репозиторий** (сейчас его нет!) — baseline, ветки, Conventional Commits.
- [ ] **Деньги → decimal** (вместо float в pv/bv) — фундаментальный фикс перед любыми финансами.
- [ ] **Sentry** (self-host sentry.adarasoft.com), **Azure tags** (project/env/owner), **Key Vault** для секретов.
- [ ] **CI/CD + контейнеризация** (ACA по нашему дефолту), `server-watchdog`.
- [ ] Решить стек на полноценный сервис (по матрице в plan-файле; дефолт — остаёмся Laravel+Next).
**Exit:** репо + пайплайн + наблюдаемость; калькулятор деплоится по кнопке.

## Фаза 1 — Доменное ядро + реальная модель сети (3–5 нед) 🧠

Превратить симулятор в переиспользуемое ядро и завести **настоящую сеть**.
- [ ] **Извлечь комп-логику** из симулятора в чистый, юнит-тестируемый домен-модуль
  (формулы/пороги/проценты как спека; покрыть тестами — сейчас их нет).
- [ ] **Нормализованная генеалогия** вместо JSON-блоба: Postgres `ltree`/closure table
  (миграция MySQL→Postgres), реальные сущности **member**, placement-дерево vs sponsorship-линия,
  конкурентная безопасность (локи/версии).
- [ ] Регистрация/размещение реальных партнёров, привязка к аккаунту (есть auth).
**Exit:** можно создать реального партнёра, поставить в сеть, ядро считает по живым данным.

## Фаза 2 — Кабинет партнёра MVP + админка MVP (4–6 нед) 👥

То, чем пользуются каждый день (главный драйвер удержания и снижения тикетов).
- **Кабинет партнёра:** дашборд дохода (разбивка по бонусам + логика расчёта),
  прогресс рангов (визуальный бар + условия), **дерево команды** (переиспользуем react-d3-tree
  из калькулятора), профиль, реф-ссылка.
- **Админка:** управление участниками, **RBAC** (роли: владелец/финансы/лидер/саппорт),
  базовые отчёты (продажи/активность/удержание), настройка бонусов/порогов.
**Exit:** партнёр видит свой реальный доход и команду; админ управляет базой и правами.

## Фаза 3 — Финансовое ядро (4–6 нед) 💰

Самое ответственное: реальные деньги.
- [ ] **Ledger с двойной записью**, идемпотентные начисления, аудит-трейл.
- [ ] **Период-клозинг** (commission run за расчётный период) поверх домен-ядра.
- [ ] **E-wallet**: баланс, история, **заявки на вывод** + одобрение в админке.
- [ ] Мультивалютные балансы (отображение уже есть).
**Exit:** автоматический прогон за период, партнёр запрашивает вывод, админ одобряет, всё в ledger.

## Фаза 4 — Commerce и платежи (4–6 нед) 🛒  *(модель A; для B — инвест-лоты/подписки)*

- [ ] Каталог, корзина, checkout, **autoship/подписки**, retry неудачных платежей (д.3/7/14),
  трекинг исполнения; связка autoship ↔ квалификация.
- [ ] Платёжные шлюзы (фиат; крипто — опц.), приём + выплаты через единый фин-интерфейс.
**Exit:** заказ → PV/объём → комиссия → кошелёк, end-to-end.

## Фаза 5 — KYC, безопасность, комплаенс (2–3 нед) 🔒

- [ ] **KYC** (Sumsub/аналог) в жизненном цикле аккаунта; **2FA**; шифрование at-rest/in-transit;
  **GDPR**-обработка; полный аудит-лог фин-операций.
**Exit:** готовность к проду с реальными деньгами и PII в юрисдикции(ях).

## Фаза 6 — Рост и вовлечённость (по мере >1000 активных) 🚀

Параллельно/после, по приоритету бизнеса:
- [ ] **Мобильное приложение** (React Native: push, алёрты по комиссиям, дерево, заказ, шеринг).
- [ ] **Уведомления** (Telegram-бот: регистрации/выплаты), массовые рассылки.
- [ ] **Replicated sites** (персональные лендинги партнёров).
- [ ] **CRM** (лиды), **ERP** (склад) интеграции.
- [ ] **Геймификация** (бейджи/лидерборды/челленджи).
- [ ] **AI-аналитика** (churn-прогноз, аномалии) — после ~6 мес живых данных.

## Сводка по этапам зрелости

| Фаза | Что даёт | Бизнес-этап |
|---|---|---|
| 0–1 | фундамент + реальная сеть | пред-MVP |
| 2 | кабинет + админка | **MVP (до 500 партнёров)** |
| 3 | деньги (ledger/выплаты) | MVP→Growth |
| 4 | commerce/платежи | Growth |
| 5 | KYC/безопасность | перед публичным запуском |
| 6 | мобайл/CRM/AI/геймификация | Growth→Scale (500–10k+) |

## Критический путь и зависимости

```
Ф0 (фундамент, decimal) → Ф1 (домен-ядро + генеалогия) → Ф2 (кабинет/админка)
                                                              ↓
                                          Ф3 (ledger/выплаты) → Ф4 (commerce/платежи)
                                                              ↓
                                                      Ф5 (KYC/безопасность) → ПРОД
                                                              ↓
                                                      Ф6 (рост: мобайл/CRM/AI)
```

- **Нельзя начинать Ф3 (деньги) без Ф0 (decimal) и Ф1 (реальная сеть).**
- Ф2 и часть Ф6 (уведомления, replicated) можно вести параллельно после Ф1.
- Калькулятор остаётся как **маркетинг/онбординг-витрина** на всём пути.

## Что решить до старта
1. **Модель бизнеса** (товарный A / инвест B) — переставляет Фазы 4–5.
2. **Стек** полноценного сервиса (дефолт — Laravel+Next; альтернативы в plan-файле).
3. **MySQL → PostgreSQL** для реальной сети/отчётов (рекомендуется; миграция в Ф1).
