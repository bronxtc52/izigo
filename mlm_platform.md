# MLM IT-платформа: каким должен быть кабинет партнёра и портал в целом

**Дата:** 2026-06-20
**Тип:** ресёрч (референс + отраслевой синтез)
**Референс:** [FlawlessMLM](https://flawlessmlm.com/en) — вендор MLM-софта с 2004 г.
(платформа Flawless Core: 40+ модулей, 400+ проектов, 5M+ партнёрских аккаунтов,
90+ рынков). Изучены страницы: `mlm-back-office-software`, `mlm-software-features`,
homepage, `5-key-features...`.

> Контекст нашего проекта: сейчас у нас (IziGo) есть только **калькулятор маркетинг-плана**
> (симулятор «что-если»: бинарное дерево, бонусы binary/leader/rank/referral, ранги, пакеты
> Bronze/Silver/Gold). Это ~1 из ~10 модулей полноценной платформы. Ниже — карта того, чем
> должна стать платформа, чтобы быть «полноценным сервисом для MLM-бизнеса».

---

## 1. Главный принцип: два портала на одной системе

Любая MLM-платформа — это **две роли в одном продукте на общей БД**:

| | Кабинет партнёра (Distributor Back Office) | Админ-портал (Admin Panel) |
|---|---|---|
| Кто | Дистрибьютор/партнёр | Компания: фин-менеджер, лидер, владелец, саппорт |
| Зачем | продаёт, рекрутирует, отслеживает доход и рост | управляет пользователями, бонусами, выплатами, отчётностью |
| Ключевое | «сколько я заработал и как расти» | «контроль бизнес-логики и денег» |

Принцип синхронности: партнёр оформил заказ → транзакция мгновенно видна в админке;
админ одобрил выплату → e-wallet партнёра обновился мгновенно. **Разные интерфейсы,
единые данные.** Хороший кабинет = меньше тикетов в саппорт (по данным референса —
сокращение обращений на 40–60%) и выше удержание (партнёр, заходящий ≥2 раз/нед, churn на 35% ниже).

---

## 2. Кабинет партнёра (Partner Back Office) — экраны и модули

Это **единственный экран, с которым партнёр работает каждый день**. Должен грузиться
быстро (sub-second), показывать свежие данные, не прятать ключевые метрики.

1. **Дашборд дохода (commission dashboard).**
   - Разбивка заработка **по типам бонусов** (binary/referral/leader/rank/…).
   - У каждой строки — **логика расчёта**, а не только сумма: какая продажа / какой партнёр
     из команды триггернул выплату.
   - История по периодам, сравнение месяц-к-месяцу.
2. **Прогресс по рангам.** Текущий ранг, условия следующего, **визуальный прогресс-бар**
   (насколько близко к следующему уровню). Пороги настроены под комп-план.
3. **Структура команды (genealogy tree).** Личное генеалогическое дерево, графический
   бинарный вид, поиск партнёра, фильтр веток; видно, где рост «стопорится» (слабая нога).
4. **Заказы и autoship.** История покупок (товар, дата, сумма, статус оплаты, трекинг
   доставки), подписки/автозаказ с датой следующего списания.
5. **E-wallet.** Баланс заработанных комиссий, **запрос на вывод**, мультивалютность
   (партнёр видит баланс в своей валюте).
6. **Обучение (Academy / training).** Пошаговые уроки с проверкой ДЗ — внутри того же
   кабинета (без отдельного логина). Снижает нагрузку на саппорт, повышает вовлечённость.
7. **Реферальные инструменты.** Персональная реф-ссылка, replicated-лендинг (см. §11),
   шаринг в соцсети/мессенджеры.
8. **Профиль и KYC-статус.** Личные данные, верификация, уведомления о квалификациях.

## 3. Админ-портал (Admin Panel) — модули

Здесь операционная команда проводит ~80% рабочего дня. Принцип: **конфигурация, а не кастом-код.**

1. **Управление участниками + KYC + роли.**
   - Полный жизненный цикл аккаунта: регистрация → одобрение → KYC (напр. Sumsub) →
     роль/статус.
   - Поиск и сегментация базы по рангу, региону, активности, дате регистрации.
   - **RBAC** (role-based access control): фин-менеджер видит выплаты; региональный лидер —
     только свою территорию; владелец — всё. Права меняются в настройках, без кода.
2. **Настройка бонусов / комп-плана.** Пороги рангов, проценты, кэпы, исключения.
3. **Одобрение выплат (payouts).** Очередь заявок на вывод, логи, сверка транзакций.
4. **Отчётность.** Реал-тайм дашборды (продажи, комиссии по рангам, активность,
   удержание), экспорт **Excel/PDF** (комплаенс, налоги, борд), фин-отчёты по периодам и KPI,
   кастомные шаблоны под регуляторику разных юрисдикций.
5. **Период-клозинг (commission run).** Запуск начисления за расчётный период с **аудит-трейлом**.
6. **Управление товаром/контентом.** Каталог, цены, контент кабинета, обучающие материалы.
7. **Коммуникации.** Массовые/таргетные уведомления (авто-триггеры при достижении ранга и т.п.).

---

## 4. Ядро: компенсационный (комиссионный) движок

Сердце платформы. Если падает — остальное не важно.
- Поддержка типов планов: **binary, unilevel, matrix, stairstep breakaway, hybrid, party plan**
  (+ generation/forced matrix). У нас сейчас — binary.
- Функции: расчёт и распределение комиссий по всей сети, **период-клозинг**, разрешение
  **квалификаций по рангам**, учёт **PV/GV**, **cap-правила**, обработка исключений.
- Требования: реал-тайм аккруал PV/GV; стабильность под пиковой нагрузкой (референс:
  2M+ узлов без задержек клозинга); прозрачность расчёта для доверия партнёров.

---

## 5. E-commerce и управление заказами

Для товарного MLM — это revenue engine.
- Каталог, корзина, checkout, **подписки/autoship**, трекинг исполнения.
- **Retry неудачных платежей** (день 3/7/14) — возвращает 8–15% сорвавшихся списаний.
- Связка autoship ↔ квалификация ранга: активный автозаказ держит ежемесячный объём.

---

## 6. Платежи и e-wallet

- **Приём** платежей (от клиентов) и **выплата** комиссий (партнёрам) — в обе стороны.
- 9+ фиатных шлюзов + крипто-шлюз (Tron/ETH/BSC/BTC).
- **Мультивалютность обязательна** при работе в >1 стране: списания/выплаты в локальной валюте,
  сверка всех транзакций в едином фин-отчёте.
- ⚠️ Деньги — только **decimal/целые «копейки»**, не float (у нашего калькулятора сейчас float —
  для реальных операций недопустимо).

---

## 7. Безопасность и комплаенс (обязательный минимум)

- RBAC, **2FA**, шифрование данных at-rest и in-transit.
- GDPR-готовность (особенно для EU), **аудит-лог** финансовых операций.
- KYC/верификация (Sumsub и аналоги), AML-сценарии.
- «Никакой список фич не компенсирует утечку данных» — безопасность до фич.

## 8. Локализация и мультивалютность

- Интерфейс на N языках (референс — 10), локаль-зависимое форматирование, отображение валют.
- Без мультиязычности закрыт доступ к быстрорастущим рынкам APAC/LatAm.
- У нас уже есть: 6 локалей (az/kk/ky/mn/ru/uz) + валюты — хороший задел.

## 9. Мобильное приложение

- **Не уменьшенная копия** десктопа, а инструмент ежедневной вовлечённости.
- Push-уведомления, алёрты по комиссиям, просмотр структуры, оформление заказа, соц-шеринг,
  регистрация рекрутов «в поле».
- Стек референса: **React Native** (iOS+Android из одной кодовой базы).

## 10. Интеграции

- CRM (лиды), платёжные шлюзы, соцсети (реф-ссылки), **ERP** (склад/фулфилмент),
  **Telegram-бот** (уведомления о регистрациях/выплатах), REST API для третьих систем.
- Важно: «заявленная» интеграция ≠ «обкатанная в проде».

## 11. Продвинутые модули (после ~1000 активных партнёров)

- **AI-аналитика:** прогноз оттока (churn), раннее выявление топ-перформеров, детект
  аномалий в комиссионных прогонах, авто-алёрт аплайну при «затухании» лидера.
  Внедрять после ~6 мес сбора данных (нужна история для обучения).
- **Геймификация:** бейджи, лидерборды, задания-челленджи, очки → обмен на мерч/билеты/бонусы.
  Лучше всего для товарного MLM с месячными циклами; для инвест/образования — вехи/сертификаты.
- **Replicated websites:** персональные лендинги/витрины партнёров под их реф-ссылкой.

## 12. Карта фич по этапам зрелости бизнеса

| Этап | Партнёров | Приоритетные модули |
|---|---|---|
| **Startup / MVP** | до 500 | комиссионный движок, кабинет партнёра, e-commerce, админка, 1 платёжный шлюз |
| **Growth** | 500–10k | мобильное приложение, CRM, геймификация, мультивалюта, autoship |
| **Scale** | 10k–100k | AI-аналитика, отчётный сьют, балансировка нагрузки, несколько шлюзов, replicated sites |
| **Enterprise** | 100k+ | предиктивные модели, выделенная инфраструктура, 24/7, региональные комплаенс-модули |

Типовая ошибка: покупать enterprise-функции на старте. **Наращивать набор под рост бизнеса.**

## 13. Архитектурные заметки (из референса)

- Frontend кабинета — **React** (sub-second загрузка); мобильный — React Native.
- Backend / БД — **PostgreSQL** (сложные отчёты ~2× быстрее MySQL; период-клозинг за минуты, не часы).
- Два портала — **единая кодовая база и БД**, разные интерфейсы.
- Событийная связность: одна продажа → комиссия → запись в кабинет → лог в CRM →
  обновление склада (one event, multiple updates, zero manual).

## 14. Привязка к IziGo: что есть и что строить

**Уже есть (наш калькулятор):**
- Компенсационная логика binary + bonuses (referral/leader/rank) + ранги + PV/GV — это
  фактически **ядро §4 в режиме симуляции** (ценный IP). Но: дерево хранится JSON-блобом,
  деньги — float, нет периодов/начислений.
- Локализация (6 языков) и мультивалюта (отображение) — задел §8.
- Локальная авторизация email+пароль (после удаления SSO).

**Нужно построить (по убыванию приоритета для «полноценного сервиса»):**
1. **Реальная модель сети** вместо JSON: нормализованная иерархия (Postgres `ltree`/closure
   table), реальные члены, размещение/спонсорство, конкурентная безопасность (локи/версии).
2. **Кабинет партнёра** (§2): дашборд дохода, прогресс рангов, дерево команды, заказы, e-wallet.
3. **Админ-портал** (§3): участники+KYC+RBAC, выплаты, отчёты, настройка бонусов, период-клозинг.
4. **Финансовое ядро**: ledger c двойной записью, идемпотентные начисления, decimal-деньги,
   расчётные периоды, clawback/возвраты.
5. **E-commerce** (§5) + **платежи/кошелёк** (§6) — если товарный/инвест MLM.
6. **Безопасность/комплаенс** (§7): 2FA, аудит, GDPR, KYC.
7. **Observability** (Sentry), CI/CD, контейнеризация — отсутствуют, обязательны до прода.
8. Далее по зрелости: мобильное приложение, интеграции, AI/геймификация, replicated sites.
